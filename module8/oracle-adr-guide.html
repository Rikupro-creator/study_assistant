What is Oracle ADR?

The Automatic Diagnostic Repository (ADR) is a centralized, file-based repository that stores diagnostic data outside of the database. This critical infrastructure component enables problem diagnosis even when the database instance is down.

Key Point: ADR is NOT held inside an Oracle database schema - it's a file system-based repository that can be accessed independently of the database state.

🗂️ File-Based Storage

Stores diagnostic data in a structured file system hierarchy, separate from the database instance.

🔍 Independent Access

Can diagnose issues even when the database instance is down or unavailable.

🎯 Multi-Product Support

Used not only for Oracle Database but also for other Oracle products like ASM, Grid Infrastructure, and Oracle Net.

Fault Diagnosability Infrastructure (FDI)

FDI is a comprehensive feature in Oracle Database that helps quickly diagnose and resolve faults. ADR is a key component of this infrastructure.

FDI Components:

  • Automatic Diagnostic Repository (ADR) - Centralized storage for all diagnostic data
  • Alert Log File - Records important database messages and alerts
  • Trace Files - Detailed technical information about errors and operations
  • Dump Files - Stack traces, memory dumps, and diagnostic data
  • Core Files - OS-generated snapshots when process crashes occur
  • Debug Log Files - Information for debugging issues and errors
  • DDL Log Files - Records Data Definition Language statements

Problems vs Incidents

📋 Problem

A problem is a critical error in the database identified by a unique problem key (error code + parameters).

ORA-00600 ORA-07445 ORA-04031

🎯 Incident

An incident is a single occurrence of a problem. Each problem can have multiple incidents associated with it.

Each incident gets a unique ID and detailed dump files.

Important: ADR is designed to handle critical incidents only. Not all database issues will be captured - some may only appear in the alert log.

ADR Directory Structure

The ADR organizes files in a hierarchical structure:

<ADR_BASE>/diag/<product_type>/<product_id>/<instance_id>

Example:
/u01/app/oracle/diag/rdbms/oradb/oradb/
    ├── alert/        (XML format alert log)
    ├── trace/        (text format alert log & trace files)
    ├── incident/     (incident dumps)
    └── cdump/        (core dumps)
Configuration: The ADR base is set via the DIAGNOSTIC_DEST parameter, which defaults to $ORACLE_BASE.

Alert Log Formats:

Format Location Purpose
XML .../alert/ Machine-readable format for automated processing
Text .../trace/alert_<SID>.log Human-readable format (preferred by DBAs)

Using ADRCI (ADR Command Interpreter)

ADRCI is the command-line tool for managing ADR. It provides powerful capabilities for viewing incidents, packaging diagnostic data, and managing file retention.

Starting ADRCI:

$ adrci

ADR base = "/u01/app/oracle"
adrci>

Essential ADRCI Commands:

View Available Homes 
adrci> show homes
adrci> show homes -rdbms    # Database homes only
Set Current Home 
adrci> set homepath diag/rdbms/oradb/oradb
View Alert Log 
adrci> show alert
adrci> show alert -tail 50
adrci> show alert -p "MESSAGE_TEXT like '%ORA-600%'"
View Incidents 
adrci> show incident
adrci> show incident -mode brief
adrci> show incident -p "incident_id=12345"
View Trace Files 
adrci> show tracefile
adrci> show tracefile -i 12345    # For specific incident
adrci> show tracefile -t          # With timestamps

Incident Packaging

When you need to send diagnostic data to Oracle Support, ADRCI can package all relevant files automatically.

Three-Step Process:

1️⃣ Create Package

adrci> ips create package

Created package 1 based on problem id 1

2️⃣ Add Incidents

adrci> ips add incident 12345 package 1

Added incident 12345 to package 1

3️⃣ Generate Physical Package

adrci> ips generate package 1 
       in /tmp

Generated file /tmp/ORA12345_20250217_1.zip

Advanced Packaging Options:

Create and Add in One Step 
# Package by problem ID
adrci> ips create package problem 1

# Package recent incidents (last 24 hours)
adrci> ips create package seconds 86400

# Package by time range
adrci> ips create package time '2025-02-17 08:00:00.00 +00:00' 
                                '2025-02-17 18:00:00.00 +00:00'

ADR Purge Policies

Oracle automatically manages ADR file retention through two policies:

⏱️ SHORTP_POLICY

Default: 720 hours (30 days)

Applies to: Trace files, core dumps, packaging information

📅 LONGP_POLICY

Default: 8760 hours (1 year)

Applies to: Incident info, incident dumps, alert logs

Viewing Current Policies:

adrci> show control

ADRID                SHORTP_POLICY        LONGP_POLICY        
-------------------- -------------------- --------------------
3705546879           720                  8760

Changing Policies:

# Change short-term policy to 48 hours
adrci> set control (SHORTP_POLICY = 48)

# Change long-term policy to 48 hours
adrci> set control (LONGP_POLICY = 48)

Manual Purging:

Note: Manual purge commands use minutes, not hours. Example: 2880 minutes = 48 hours 
# Purge by age (2880 minutes = 48 hours)
adrci> purge -age 2880 -type INCIDENT
adrci> purge -age 2880 -type ALERT
adrci> purge -age 2880 -type TRACE
adrci> purge -age 2880 -type CDUMP
adrci> purge -age 2880 -type HM

# Apply current policy settings immediately
adrci> purge
Important: ADR purge policy is automatically applied to the database home but NOT to other components (listener, ASM, Grid Infrastructure). You must apply purge policies manually for these components.

Key Features of ADR

📊 Centralized Repository

All diagnostic data (logs, traces, dumps, alerts) stored in a single, organized location.

🤖 Automatic Organization

Data automatically organized by timestamp, component, and incident for easy analysis.

🔧 Multi-Component Support

Supports diagnostics for database, Oracle Net, ASM, and other Oracle products.

📦 Incident Packaging

Built-in tools to package diagnostic data for Oracle Support using ADRCI.

♻️ Retention Policy

Automatically purges old diagnostic data based on configurable retention settings.

🔄 Health Monitoring

Integrates with Health Monitor to provide proactive health checks and recommendations.

Best Practices

  • Regular Monitoring: Check ADR regularly for new incidents and problems
  • Adjust Retention: Modify purge policies based on storage constraints and compliance requirements
  • Archive Important Incidents: Package and archive critical incidents before they're purged
  • Monitor Disk Space: Ensure adequate space for ADR, especially on busy systems
  • Automate Purging: Schedule regular purge operations for non-database components
  • Review Alert Logs: Not all issues create incidents - review alert logs for warnings
  • Use Web Search Tools: Leverage ADRCI's search capabilities to find specific errors quickly
  • Document Incidents: Maintain a log of resolved incidents for future reference
