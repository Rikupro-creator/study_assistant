Key Components

Redo Log Buffer

Memory area in the primary instance where redo entries are initially written before being flushed to disk or shipped to standby.

Log Writer (LGWR)

Background process that flushes redo log entries from the buffer to online redo log files in cyclic order.

Archiver (ARC)

Copies filled online redo log files to archive log destination for backup and recovery purposes.

Log Shipping Service (LNS)

The LNS (Log Network Service / Log Writer Network Service) is the critical component responsible for shipping redo data from primary to standby.

Primary Responsibilities

  • Read Redo Log Buffer: Directly reads redo entries from the primary database's redo log buffer
  • Network Transmission: Ships redo data over Oracle Net Services to the standby database
  • Adaptive Behavior: Can switch to reading from online redo log files if buffer read falls behind
Performance Consideration: During high transaction volumes, if LNS cannot keep pace with LGWR, it automatically switches to reading from online redo log files instead of the redo log buffer. This ensures no data loss while maintaining performance.
Network Configuration: LNS requires properly configured Oracle Net Services between primary and standby servers—identical to client/server networking setup but between database instances.

Remote File Service (RFS)

The RFS (Remote File Service / Remote Fetch Service) operates on the standby server and handles incoming redo data.

Core Functions

🔄 Receive Redo Data

Accepts redo log entries transmitted by LNS from the primary database over the network.

📝 Write to SRL

Writes received redo entries to Standby Redo Log (SRL) files on the standby database.

🔍 Gap Detection

Automatically identifies sequence number gaps between primary and standby databases.

🩹 Gap Resolution

Proactively fetches missing archive log files from the primary to resolve gaps automatically.

Automatic Gap Resolution

Gap Detection

RFS compares current standby sequence number with primary sequence number (e.g., Primary: 50, Standby: 25 = Gap of 25 sequences)

Archive Fetch

RFS contacts the primary's ARC (Archiver) process and requests missing archive log files

Transfer & Apply

Missing archives are transferred to standby, then MRP applies them to synchronize the databases

Manual Intervention Required: If RMAN has already backed up and deleted archive logs on the primary before they were shipped to standby, automatic gap resolution will fail. DBAs must manually restore and transfer the required archives.

Standby Redo Logs (SRL)

Standby Redo Logs are special redo log files that exist separately from standard redo log files and play a crucial role in Data Guard architecture.

SRL vs Standard Redo Logs

Aspect Standard Redo Logs Standby Redo Logs (SRL)
Purpose Record current database operations Receive redo data from primary database
Active When Database is open and running Database is in standby mode receiving logs
Written By LGWR (Log Writer) RFS (Remote File Service)
Read By ARC (for archiving) MRP (for recovery)
Configuration Always present in all databases Must be manually configured for Data Guard
Critical Configuration Requirement:

SRLs must be configured on BOTH primary and standby databases. This is essential for switchover and failover operations.

Why? During role reversal (switchover/failover), the current primary becomes the new standby. Without SRLs on the original primary, it cannot function as a standby after the role change.

When Standard Redo Logs Become Active

On a standby database, standard redo log files remain inactive while the database is in managed recovery mode (mount mode). They become active only when:

  • The standby is opened for read/write operations
  • A switchover or failover promotes the standby to primary
  • The database role changes from standby to primary

Managed Recovery Process (MRP)

The MRP (Managed Recovery Process) is the log apply service that keeps the standby database synchronized with the primary.

How MRP Works

📖 Read SRL Files

Continuously reads redo entries from Standby Redo Log files as they are written by RFS.

⚙️ Apply Changes

Applies the redo changes to the physical standby database, keeping it synchronized block-by-block.

🔄 Continuous Operation

Runs continuously in the background while the standby database is in managed recovery mode.

📊 Progress Tracking

Maintains metadata about applied sequences, enabling gap detection and synchronization monitoring.

Block-Level Replication: MRP ensures the standby database is an exact physical copy of the primary, replicating changes at the block level. This guarantees data consistency and enables fast failover capabilities.

Failover & Switchover Requirements

Proper configuration is essential for successful role transitions in Data Guard environments.

Critical Configuration Checklist

  • SRL on Primary: Standby Redo Logs must be configured on the primary database
  • SRL on Standby: Standby Redo Logs must be configured on the standby database
  • Network Connectivity: Bidirectional Oracle Net Services must be configured and tested
  • Archive Log Destination: LOG_ARCHIVE_DEST_n parameters must point to both local and remote destinations
  • Parameter Files: Both databases need compatible initialization parameters
Common Mistake:

Many DBAs configure SRLs only on the standby database, assuming they're only needed there. This prevents successful switchover/failover operations when the primary needs to become a standby.

Impact: Without SRLs on the original primary, role reversal fails, potentially causing extended downtime during disaster recovery scenarios.

What Happens During Switchover

Primary → Standby Conversion

Original primary transitions to standby role. LNS stops, RFS starts accepting logs from new primary

-- Primary becomes standby
-- LGWR stops writing to standard redo logs
-- RFS starts, writes to SRLs
-- MRP begins applying received logs
Standby → Primary Conversion

Original standby promotes to primary role. MRP stops, LGWR activates standard redo logs

-- Standby becomes primary
-- MRP completes final log application
-- Database opens READ WRITE
-- LGWR activates standard redo logs
-- LNS starts shipping to new standby

RMAN-08137 Error

Error Message: 
RMAN-08137: WARNING: archived log not deleted, needed for 
standby or upstream capture process

This warning appears when RMAN refuses to delete archive logs because they are still required by standby databases or capture processes like GoldenGate. If ignored, this can rapidly fill archive log storage and threaten database availability.

Root Causes

📡 Standby Lag

Archive logs haven't been applied on standby database yet. Primary retains them until standby confirms application.

🔄 GoldenGate/Streams

Upstream capture processes (GoldenGate Extract, Streams) still need these logs for replication.

👻 Orphaned Processes

Registered capture processes remain in database metadata even after decommissioning, blocking deletion.

⚙️ Policy Misconfiguration

Archive log deletion policy set too conservatively (waiting for APPLIED vs SHIPPED status).

Diagnosing RMAN-08137

Method 1: Check Archive Log Dependencies

Check for Data Guard Gaps 
-- Identify missing sequences on standby
SELECT THREAD#, LOW_SEQUENCE#, HIGH_SEQUENCE# 
FROM V$ARCHIVE_GAP;

Interpretation: Results show sequence number ranges missing on standby. These archives must stay on primary until gap resolves.

Review Retained Archives 
-- List all archives not yet deleted
SELECT SEQUENCE#, FIRST_TIME, NEXT_TIME 
FROM V$ARCHIVED_LOG 
WHERE DELETED = 'NO' 
ORDER BY SEQUENCE#;

Action: Identify which files are held and when they were created. Cross-reference with V$ARCHIVE_GAP to determine dependency.

Check Deletion Policy 
-- View current RMAN configuration
RMAN> SHOW ALL;

Look for: APPLIED ON STANDBY or SHIPPED TO STANDBY policies. These keep archives until standby confirms receipt/application.

Temporary Policy Adjustment

-- Change to less restrictive policy (if safe)
RMAN> CONFIGURE ARCHIVELOG DELETION POLICY TO SHIPPED TO STANDBY;

-- Attempt deletion
RMAN> DELETE NOPROMPT ARCHIVELOG ALL COMPLETED BEFORE 'SYSDATE-1';

-- Restore original policy
RMAN> CONFIGURE ARCHIVELOG DELETION POLICY TO APPLIED ON STANDBY;
Caution: Never bypass deletion policies without verifying standbys and capture processes are healthy. Premature deletion can break replication or prevent point-in-time recovery.

Method 2: Capture Processes & Standby Status

Check Registered Capture Processes

-- List all capture processes
SELECT CAPTURE_NAME, STATUS FROM DBA_CAPTURE;

Common Issue: Decommissioned GoldenGate extracts or Streams captures remain registered even after being stopped, continuing to block archive log deletion.

Unregister Unused Captures (GoldenGate)

-- In GoldenGate GGSCI console
GGSCI> DBLOGIN USERIDALIAS 
GGSCI> UNREGISTER EXTRACT  DATABASE

Monitor Standby Apply Status

-- Check recovery processes on standby
SELECT PROCESS, STATUS, THREAD#, SEQUENCE#, BLOCK#, BLOCKS 
FROM V$MANAGED_STANDBY 
WHERE PROCESS IN ('MRP0', 'RFS');

High Apply Lag: If MRP shows large sequence number differences from primary, standbys need time to catch up before archives can be safely deleted.

Check Archive Destination Status

-- Verify standby synchronization
SELECT DEST_ID AS "Dest", 
       STATUS AS "Status", 
       APPLIED_SCN AS "Applied SCN" 
FROM V$ARCHIVE_DEST_STATUS 
WHERE STATUS = 'VALID';
Compare with Current SCN 
-- Get current primary SCN
SELECT CURRENT_SCN FROM V$DATABASE;

Large SCN differences indicate apply lag on standby destinations.

Method 3: Proactive Prevention

Prevent RMAN-08137 errors through continuous monitoring and automation rather than reactive troubleshooting.

Beginner Level: Monitor Disk Space

-- Check recovery file destination space
SELECT NAME AS "Archive Dest", 
       SPACE_LIMIT - SPACE_USED AS "Free Space MB" 
FROM V$RECOVERY_FILE_DEST;

Set Alerts: Configure Enterprise Manager or custom scripts to alert when free space drops below 20% threshold.

Intermediate Level: Automated Reporting

-- Daily report of un-applied sequences
SELECT d.DEST_NAME, 
       d.STATUS, 
       l.SEQUENCE# AS "Last Applied",
       (SELECT MAX(SEQUENCE#) FROM V$ARCHIVED_LOG) AS "Current",
       (SELECT MAX(SEQUENCE#) FROM V$ARCHIVED_LOG) - l.SEQUENCE# AS "Gap"
FROM V$ARCHIVE_DEST d
LEFT JOIN V$ARCHIVED_LOG l ON d.DEST_ID = l.DEST_ID 
                          AND l.APPLIED = 'YES'
WHERE d.STATUS = 'VALID'
ORDER BY d.DEST_ID;

Advanced Level: Automated Alerting

  • Apply Lag Monitoring: Track MRP apply lag continuously, alert when exceeding thresholds
  • Email/SMS Integration: Connect Oracle alerting to external monitoring tools for immediate notification
  • Capture Process Audits: Schedule weekly reviews of DBA_CAPTURE to identify orphaned entries
  • Archive Retention Metrics: Dashboard showing archive age, count, and storage consumption trends
  • Automated Gap Resolution: Scripts to detect and resolve gaps before manual intervention needed
Best Practice: Combine multiple monitoring layers. Disk space alerts catch immediate problems, while apply lag trends predict future issues before they become critical.

Gap Resolution Scenarios

Scenario 1: Automatic Resolution (Happy Path)

Condition: All required archive logs exist on primary after sequence 25

Result: RFS automatically contacts ARC, fetches missing archives, and MRP applies them

Scenario 2: RMAN Backup Conflict

Problem: RMAN backed up and deleted archives 26-35 before they shipped to standby

Resolution Required: Manual intervention needed

Restore Missing Archives 
-- On primary, restore archives from RMAN backup
RMAN> RESTORE ARCHIVELOG FROM SEQUENCE 26 UNTIL SEQUENCE 35;
Manual Transfer 
-- Copy restored archives to standby
$ scp arch_26_*.arc standby:/arch_dest/
Register & Apply 
-- On standby, register archives
ALTER DATABASE REGISTER PHYSICAL LOGFILE '/arch_dest/arch_26_*.arc';

-- Verify MRP picks them up
SELECT SEQUENCE#, APPLIED FROM V$ARCHIVED_LOG 
WHERE SEQUENCE# BETWEEN 26 AND 35;

Scenario 3: Network Issues

Symptoms: LNS errors, RFS connectivity failures, growing gap

Actions:

  • Test Oracle Net connectivity: tnsping STANDBY_TNSALIAS
  • Check listener status on standby: lsnrctl status
  • Review alert logs for ORA-12541, ORA-12170 errors
  • Verify firewall rules allow database port traffic

Best Practices Summary

✅ Always Configure SRLs

Set up Standby Redo Logs on both primary and standby from day one, even if failover isn't immediately planned.

✅ Monitor Apply Lag Daily

Check V$MANAGED_STANDBY and V$ARCHIVE_GAP daily. Address growing gaps before they become critical.

✅ Right-Size Deletion Policies

Balance between "APPLIED" (safer) and "SHIPPED" (less storage) based on recovery objectives and disk capacity.

✅ Test Switchover Regularly

Quarterly switchover tests validate configuration, train staff, and identify issues before real disasters.

✅ Clean Up Captures

Audit DBA_CAPTURE quarterly. Unregister unused GoldenGate/Streams processes immediately after decommissioning.

✅ Document Dependencies

Maintain runbook documenting all Data Guard destinations and capture processes consuming archives.

Remember: Most Data Guard issues stem from incomplete knowledge and missing queries. Master the diagnostic views (V$ARCHIVE_GAP, V$MANAGED_STANDBY, DBA_CAPTURE) and most problems become straightforward to solve.
