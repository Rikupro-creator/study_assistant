🔐 Authentication Methods in Oracle

Authentication = verifying the identity of a user, program, or device.

Main methods:

  • Oracle Database authentication: username/password stored inside the database.
  • Operating System (OS) authentication: user already authenticated by OS; no password required for DB login.
  • Password file authentication: credentials stored in an external password file.
  • Network-based methods: SSL, directory-based, third-party.
📌 OS Authentication: Two types – administrative (for DBAs) and non-administrative (normal users). This lecture focuses on OS authentication.

🧭 Administrator Authentication Flowchart

The logic Oracle follows when an admin connects:

┌─────────────────────────────────┐ │ Administrator wants to connect │ └──────────────┬──────────────────┘ ▼ ┌─────────────────────────────┐ │ Connection method? │ ├─────────────┬───────────────┤ │ Local (no @)│ Remote (with @)│ └──────┬──────┴───────┬───────┘ ▼ ▼ ┌──────────────────┐ ┌──────────────────┐ │ Can use OS auth │ │ Secure network? │ │ OR password file │ └───────┬──────────┘ └──────┬───────────┘ ▼ ▼ ┌──────────────────┐ ┌──────────────────┐ │ YES (e.g. SSL) │ ┌──────────────────┐ │ OS auth: │ └────────┬─────────┘ │ NO (non-secure) │ │ checks group │ ▼ └────────┬─────────┘ │ membership. │ ┌──────────────────┐ ▼ │ Password ignored │ │ Can use OS auth │ ┌──────────────────┐ └──────────────────┘ │ OR password file │ │ MUST use │ └──────────────────┘ │ password file │ └──────────────────┘
⚠️ Critical: A local user who is a member of the admin group (e.g., dba) can connect as SYSDBA with an incorrect password! OS authentication overrides password check locally.

If no password file exists, only OS authentication is possible (for local/secure connections).

🔧 Adding a user to an OS group (Linux):
sudo usermod -aG dba username # Adds 'username' to dba group

⚙️ Configuring Groups with DBCA

The DBCA utility allows you to choose which OS group maps to each administrative privilege. Groups must exist before running DBCA.

  • Typical small environment: Use a single group (dba) for all privileges.
  • Large environment (separation of duties): Create distinct groups (e.g., backupdba for backup team).

Example: If the OS group backupdba exists, DBCA can assign it to OSBACKUPDBA.

🔌 Connecting via OS Authentication (SQL*Plus)

Local OS authentication: No @ symbol used.

-- Connect as SYSDBA using OS authentication (requires membership in dba group)
sqlplus / as sysdba

-- Connect as SYSBACKUP using OS authentication (requires membership in backupdba group)
sqlplus / as sysbackup

Windows note: You must specify the net service name, e.g., sqlplus /@orcl as sysdba.

🚨 Security Concern: A system administrator can add themselves to the dba group and gain SYSDBA access without a password. In sensitive environments, separate OS and DBA roles or use additional controls.

🧑‍💻 OS-Authenticated Normal Users

Normal OS users can also connect without a password, but a corresponding database user must exist with the IDENTIFIED EXTERNALLY clause and a specific prefix.

Procedure (non-CDB):

  1. Create the OS user (e.g., user1).
  2. Check OS_AUTHENT_PREFIX (default is OPS$).
    3. SHOW PARAMETER os_authent_prefix
    -- Default: OPS$
  3. Create database user with the prefix + OS username, and IDENTIFIED EXTERNALLY.
    4. CREATE USER ops$user1 IDENTIFIED EXTERNALLY;
    GRANT create session TO ops$user1;
  4. Connect: OS user logs into the server, sets environment, and runs sqlplus /.
⚠️ Best practice: Avoid OS-authenticated users in production unless absolutely necessary.

🧩 Multitenant Challenges

Creating OS-authenticated common users is tricky:

  • Common users must start with C## (value of COMMON_USER_PREFIX).
  • OS_AUTHENT_PREFIX defaults to OPS$.
  • Linux does not allow # in usernames.

Possible (but not recommended) workaround: Change both prefixes to a compatible value (e.g., ZZ).

ALTER SYSTEM SET os_authent_prefix='' SCOPE=SPFILE;
ALTER SYSTEM SET common_user_prefix='ZZ' SCOPE=SPFILE;
-- Then create OS user 'zzuser1' and DB user 'zzuser1' IDENTIFIED EXTERNALLY

Recommendation: Avoid OS authentication in multitenant; keep defaults.

📁 What is the Password File?

An external file storing passwords of users with administrative privileges. Used when OS auth is not possible (e.g., remote non-secure connections).

  • SYS password is only in the password file (since 12.2). Previously, it was also in the dictionary.
  • Other admin users (e.g., SYSBACKUP) have passwords in both the password file and the data dictionary.
  • One password file per database.

Default locations:

-- Linux (under ORACLE_HOME/dbs)
$ORACLE_HOME/dbs/orapw<SID>

-- Windows (under ORACLE_HOME\database)
%ORACLE_HOME%\database\PWD<SID>.ora

Query current password file info:

SELECT * FROM v$passwordfile_info;
SELECT * FROM v$pwfile_users; -- Lists users with admin privileges in the password file

🛠️ Creating a Password File with orapwd

Oracle utility orapwd creates or modifies the password file.

Basic syntax (no spaces around =):

orapwd FILE='$ORACLE_HOME/dbs/orapwORCL' \
FORCE=y \
FORMAT=12.2 \
SYS='password' # or omit to be prompted
  • FILE – mandatory, full path and name (must match expected format).
  • FORCE=y – overwrite existing file.
  • FORMAT – 12 or 12.2 (default, recommended).
# Example: create password file for SID 'orcl', will prompt for SYS password
orapwd FILE='$ORACLE_HOME/dbs/orapworcl' FORMAT=12.2

⚙️ Parameter: REMOTE_LOGIN_PASSWORDFILE

ValueDescription
EXCLUSIVE (default)Password file used by one instance only. Allows adding/removing users.
SHAREDMultiple instances can share. File is read-only; cannot add users or change passwords.
NONEPassword file authentication disabled. No remote admin via password file.

In practice, keep the default EXCLUSIVE unless you have a specific need for SHARED (e.g., RAC).

🥊 OS Authentication vs. Password File

When both are applicable, OS authentication takes precedence.

-- Local connection with wrong password, but OS user is in dba group
[oracle@server]$ sqlplus sys/wrong_pass as sysdba
-- Connected. (OS auth used, password ignored)
-- Remote (network) connection with @, even from same machine
[oracle@server]$ sqlplus sys/wrong_pass@orcl as sysdba
-- ERROR: ORA-01017: invalid username/password; logon denied

The @ forces a network connection, disabling OS authentication. The password must be correct (password file is used).

🔄 Recovering from a Lost Password File

If the password file is lost or corrupted:

  1. Create a new password file using orapwd. You can set any temporary SYS password.
  2. Find users with admin privileges (they are still in the data dictionary).
    3. SELECT username, sysdba, sysoper, sysbackup FROM v$pwfile_users;
  3. Resynchronize: For each such user, revoke and re-grant the administrative privilege.
    4. REVOKE sysbackup FROM sysuser1;
    GRANT sysbackup TO sysuser1;

    This action writes the user's current password (from the dictionary) into the new password file.

Does not affect OS-authenticated users.

🔒 Protecting the Password File

Default permissions (Linux):

-rw------- 1 oracle oinstall 7680 Mar 10 15:22 orapworcl

Only the Oracle software owner (and root) can read/write. Do not change these permissions.
