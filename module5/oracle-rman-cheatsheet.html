🔷PDB Backup

From ROOT (Multiple PDBs)

BACKUP PLUGGABLE DATABASE pdb1, pdb2;

From PDB (Single PDB)

BACKUP DATABASE;

🔷Tablespace Backup

Local Tablespace

When connected to the container:

BACKUP TABLESPACE tbs1;

PDB Tablespace from ROOT

🔧Recovery Procedures

🔷Complete CDB Recovery

Step-by-Step Recovery Process

  1. STARTUP MOUNT;
  2. RESTORE DATABASE;
  3. RECOVER DATABASE;
  4. ALTER DATABASE OPEN;
  5. ALTER PLUGGABLE DATABASE ALL OPEN;

🔷ROOT Container Recovery

⚠️ Important: Double quotes are required when specifying "ROOT"
RESTORE DATABASE ROOT; RECOVER DATABASE ROOT;

🔷PDB Recovery

From ROOT (Multiple PDBs)

RESTORE PLUGGABLE DATABASE pdb1, pdb2; RECOVER PLUGGABLE DATABASE pdb1, pdb2;

From PDB (Single PDB)

RESTORE DATABASE; RECOVER DATABASE;

🔷Individual Datafile Recovery

Datafile Recovery Workflow

  1. ALTER DATABASE DATAFILE '/path/file.dbf' OFFLINE;
  2. RESTORE DATAFILE '/path/file.dbf';
  3. RECOVER DATAFILE '/path/file.dbf';
  4. ALTER DATABASE DATAFILE '/path/file.dbf' ONLINE;

🔷Seed PDB Recovery

RESTORE PLUGGABLE DATABASE "PDB$SEED"; RECOVER PLUGGABLE DATABASE "PDB$SEED";

🔷Point-in-Time Recovery (PITR)

PITR Workflow

  1. ALTER PLUGGABLE DATABASE pdb1 CLOSE;
  2. RESTORE PLUGGABLE DATABASE pdb1 UNTIL TIME '2024-01-15:10:00:00';
  3. RECOVER PLUGGABLE DATABASE pdb1 UNTIL TIME '2024-01-15:10:00:00';
  4. ALTER PLUGGABLE DATABASE pdb1 OPEN RESETLOGS;

⚠️Key Restrictions

🚫 From PDB: Cannot backup, restore, or delete archived redo logs. You must connect to ROOT for these operations.

⏮️Flashback Database Overview

Purpose: Allows reverting the database to a previous state, like "rewinding a tape," without restoring from backups. Perfect for undoing mistakes such as failed application schema upgrades.

🔷How It Works

Oracle Creates Flashback Logs
→
RVWR Process Manages Logs
→
Circular Storage in FRA
→
Restore + Forward Recovery

Flashback Logs

Created in the Fast Recovery Area (FRA) with block images stored sequentially in circular fashion (not archived).

RVWR Process

Background Flashback Writer process manages log creation and maintenance automatically.

🔷Setup & Configuration

Requirements & Setup Steps

  1. Enable ARCHIVELOG mode (prerequisite)
  2. Configure Fast Recovery Area (FRA) with sufficient space
  3. Set DB_FLASHBACK_RETENTION_TARGET (e.g., 2880 minutes = 2 days)
  4. Enable Flashback Database: ALTER DATABASE FLASHBACK ON;
  5. Verify status: SELECT FLASHBACK_ON FROM V$DATABASE;
Note: DB_FLASHBACK_RETENTION_TARGET is a target, not a guarantee. Actual retention depends on FRA space availability.

🔷Flashback Operations

State Requirement: Database must be in MOUNT state to perform flashback operations.

Flashback Methods

You can flashback using:

Time SCN Sequence Number Restore Point

Inclusive (Default)

FLASHBACK DATABASE TO SCN 23565;

Includes the specified SCN/time in the recovery.

Exclusive

FLASHBACK DATABASE TO BEFORE SCN 23565;

Excludes the specified SCN/time from recovery.

Post-Flashback Verification

  1. Open database in READ ONLY mode to verify data
  2. Review and confirm the flashback was successful
  3. Open in READ/WRITE mode with RESETLOGS

🔷PDB Flashback

PDB Flashback Process

  1. Connect to CDB root as SYSDBA or SYSBACKUP
  2. Close the target PDB
  3. Run FLASHBACK PLUGGABLE DATABASE command
  4. Open the PDB with RESETLOGS

🔷Considerations & Limitations

⚠️ Cannot Flashback:
  • Dropped datafiles
  • Resized files (must be offline if shrunk)
✅ Supported:
  • Auto-extensible files
  • Flashback to BEFORE RESETLOGS

🔷Monitoring Tools

V$FLASHBACK_DATABASE_LOG

Provides estimated flashback size, oldest SCN, and oldest flashback time available.

V$FLASHBACK_DATABASE_STAT

Shows hourly flashback data, redo data, and block activity statistics.

🔷Best Practices

  • Enable in Production: Always enable Flashback Database for production environments
  • Monitor FRA Space: Regularly check Fast Recovery Area space utilization
  • Short-term vs Long-term: Use Flashback for short-term recovery, RMAN PITR for long-term
  • Watch Flashback Logs: Monitor log generation and retention target alignment
  • Restore Points: Avoid keeping unnecessary guaranteed restore points to prevent FRA overflow

📍Restore Points

Purpose: Bookmark specific SCNs for easy flashback operations without remembering exact numbers or timestamps.

🔷Types of Restore Points

Normal Restore Point

Lightweight Auto-Managed

Automatically deleted when retention period expires or space is needed.

CREATE RESTORE POINT rp_name;

Guaranteed Restore Point

Manual Delete Space Protected

Never deleted automatically. Ensures flashback availability even without enabled logging.

CREATE RESTORE POINT rp_name GUARANTEE FLASHBACK DATABASE;

🔷Commands

Create Normal Restore Point

CREATE RESTORE POINT rp_name;

Create Guaranteed Restore Point

CREATE RESTORE POINT rp_name GUARANTEE FLASHBACK DATABASE;

Drop Restore Point

DROP RESTORE POINT rp_name;
Limitations: Restore points have the same limitations as Flashback Database operations (dropped datafiles, shrunk datafiles, etc.)

🕐Point-in-Time Recovery (PITR) Overview

Purpose: Allows restoring the database (or parts of it) to a previous state when data was accidentally changed or required archive logs are missing.

🔷Levels of PITR

Database Level (DBPITR)

Full database recovery to a specific point in time.

Tablespace Level (TSPITR)

Recovery of specific tablespaces independently.

Table Level

Recover individual tables (advanced feature).

Recovery Points: Can specify using SCN, redo log sequence number, restore point, or timestamp. Oracle internally converts all to SCN.

🔷PITR vs Flashback Database

Flashback Database

Easier Faster Limited Retention

Constrained by FRA space and retention target. Best for recent changes.

DBPITR

Flexible Long History Slower

Can go as far back as backups allow, but requires downtime and is time-consuming.

🔷Requirements for DBPITR

  • ARCHIVELOG Mode: Database must be in ARCHIVELOG mode
  • Complete Backups: Backups of all datafiles prior to the target recovery SCN must exist

🔷RESETLOGS and Incarnations

What RESETLOGS Does

  • Archives current online redo logs (if present)
  • Clears online redo logs and resets log sequence to 1
  • Updates data files, redo logs, and archive logs with a new RESETLOGS SCN
  • Required after DBPITR operations
Database Incarnation: Every RESETLOGS operation generates a new incarnation. Oracle matches archived redo log SCN with database incarnation SCN during recovery. This is crucial for restoring backups from the correct incarnation.

🔷DBID Considerations

DBID Required When: Restoring control files or SPfiles from RMAN backups.

Methods to Get DBID:
  • Check RMAN log files from past backup jobs
  • Parse AUTOBACKUP control file name (DBID encoded in filename)
  • Best Practice: Save DBID externally when taking backups

🔷DBPITR Procedure

Complete DBPITR Workflow

  1. Determine recovery point (SCN, timestamp, log sequence, or restore point)
  2. Shutdown database
  3. Startup in NOMOUNT mode (if restoring control file)
  4. Set DBID (if necessary for control file restore)
  5. Restore control file from latest AUTOBACKUP
  6. Mount database
  7. RESTORE DATABASE – restore data files from backups
  8. RECOVER DATABASE – apply redo logs to reach recovery point
  9. (Optional) OPEN READ ONLY to verify data before RESETLOGS
  10. OPEN DATABASE WITH RESETLOGS – start new incarnation
⚠️ Caution: Restoring the control file overwrites the current control file. This modified procedure is useful when the exact recovery point is unclear and multiple attempts may be needed.

🔷Key Notes

  • Downtime Required: DBPITR makes the database unavailable during recovery
  • Time-Consuming: Requires restoring and recovering all data files
  • Alternative: Flashback Database can often be used as a faster alternative for recent changes

💡Important Notes & Best Practices

🔷Backup Strategy

3-2-1 Backup Rule:
  • 3 copies of your data (1 primary + 2 backups)
  • 2 different media types (disk, tape, cloud)
  • 1 offsite backup for disaster recovery

Retention Policy

Configure retention policy to automatically manage backup lifecycle:

CONFIGURE RETENTION POLICY TO RECOVERY WINDOW OF 7 DAYS;

Backup Compression

Enable compression to save storage space (typically 2-3x reduction):

CONFIGURE COMPRESSION ALGORITHM 'MEDIUM';

Backup Validation

Regularly validate backups to ensure they're usable:

VALIDATE BACKUPSET ALL;

Control File Autobackup

Essential for complete recovery scenarios:

CONFIGURE CONTROLFILE AUTOBACKUP ON;

🔷Connection Best Practices

Authentication Methods:
  • SYSDBA: Full administrative privileges (can perform all operations)
  • SYSBACKUP: Dedicated backup/recovery role (recommended for separation of duties)
  • SYSDG: For Data Guard operations
-- Connect to CDB ROOT RMAN TARGET "sys@cdb_root AS SYSDBA" -- Connect to specific PDB RMAN TARGET "sys@pdb1 AS SYSDBA" -- Using SYSBACKUP (recommended) RMAN TARGET "backup_user@cdb_root AS SYSBACKUP"

🔷Performance Optimization

Parallelism

Speed up backups with multiple channels:

CONFIGURE DEVICE TYPE DISK PARALLELISM 4;

Block Change Tracking

Dramatically speeds up incremental backups:

ALTER DATABASE ENABLE BLOCK CHANGE TRACKING;
⚠️ Performance Considerations:
  • Run backups during off-peak hours when possible
  • Monitor I/O impact on production workload
  • Use incremental backups to reduce backup time and storage
  • Consider backup network bandwidth for network-attached storage

🔷Incremental Backup Strategy

Incremental Backup Levels:
  • Level 0: Full backup (base for incremental strategy)
  • Level 1 Differential: Backs up blocks changed since the last Level 0
  • Level 1 Cumulative: Backs up blocks changed since the last Level 0 or Level 1
-- Weekly Level 0 backup BACKUP INCREMENTAL LEVEL 0 DATABASE PLUS ARCHIVELOG; -- Daily Level 1 incremental backup BACKUP INCREMENTAL LEVEL 1 DATABASE PLUS ARCHIVELOG;

🔷Testing & Validation

🔥 Critical: A backup is only as good as your ability to restore it!
  • Test restores regularly (monthly recommended)
  • Document recovery procedures and keep them updated
  • Practice emergency scenarios with your team
  • Measure RTO/RPO (Recovery Time/Point Objectives)
  • Use RESTORE... PREVIEW to check what will be restored
-- Preview restore without actually restoring RESTORE DATABASE PREVIEW; -- Validate specific backup pieces RESTORE DATABASE VALIDATE;

🔷Monitoring & Maintenance

Check Backup Status

LIST BACKUP SUMMARY; LIST BACKUP OF DATABASE; REPORT OBSOLETE;

View Backup History

SELECT * FROM V$BACKUP_SET; SELECT * FROM V$BACKUP_PIECE; SELECT * FROM V$BACKUP_FILES;

Cleanup Old Backups

DELETE OBSOLETE; DELETE EXPIRED BACKUP;

Crosscheck Backups

CROSSCHECK BACKUP; CROSSCHECK ARCHIVELOG ALL;

🔷Common Pitfalls to Avoid

⚠️ Common Mistakes:
  • Not testing restores: Don't assume backups work—test them!
  • Insufficient FRA space: Monitor and size appropriately for Flashback logs
  • Missing archive logs: Ensure archive log retention covers backup windows
  • Forgetting control file backups: Always enable AUTOBACKUP
  • Not documenting DBID: Save it before you need it!
  • Ignoring backup failures: Set up alerts and monitor logs daily
  • Not securing backup files: Encrypt sensitive data at rest and in transit
  • Wrong recovery point: Verify SCN/timestamp before starting PITR

🔷Security Considerations

Backup Security Best Practices:
  • Enable encryption: Use TDE or RMAN encryption for backups
  • Secure backup location: Restrict access to backup files and directories
  • Use SYSBACKUP: Separate backup operations from full DBA privileges
  • Audit backup operations: Track who performs backup/recovery operations
  • Offsite backups: Store copies offsite for disaster recovery
-- Enable backup encryption CONFIGURE ENCRYPTION FOR DATABASE ON; SET ENCRYPTION ON IDENTIFIED BY "password";

🔷Recovery Time Optimization

Strategies to Reduce Recovery Time:
  • Use Flashback Database for fast recovery of recent changes
  • Implement incremental backups to reduce restore time
  • Keep online redo logs properly sized and on fast storage
  • Use parallelism during restore and recovery operations
  • Maintain archived redo logs on fast, accessible storage
  • Consider standby database for critical systems (Data Guard)

🔷Documentation Requirements

Essential Documentation to Maintain

  • DBID - Save immediately after database creation
  • Backup schedules - Full, incremental, archive log policies
  • Retention policies - How long backups are kept
  • Recovery procedures - Step-by-step for different scenarios
  • Network paths - Backup destinations and mount points
  • Contact information - On-call DBAs, management, vendors
  • RTO/RPO objectives - Business requirements for recovery
  • Encryption keys - Securely stored wallet passwords
