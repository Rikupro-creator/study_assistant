What is RMAN?

Recovery Manager (RMAN): Oracle's built-in utility for database backup and recovery.

✅
Included with Oracle Standard and Enterprise Editions

No additional license required

💻
Runs outside the database

But integrates seamlessly with it

📝
Command-line interface

With scripting capabilities

Key Benefits

⚡ Efficient Backups

Skips unused data blocks to save time and space

📈 Incremental Backups

Only backs up changes since the last backup

🌍 Platform Independent

Works across different operating systems

🔒 Encryption

Secures backup files

🔄 Database Cloning

Create copies offline or online

📼 Tape Compatibility

Works with Oracle Secure Backup, NetBackup, TSM, etc.

☁️ Cloud Support

Direct backups to Oracle Cloud or Amazon S3

🗜️ Compression

Reduces backup file sizes

🔍 Corruption Detection

Detects corrupted blocks during backup

RMAN Command Language Overview

Key Concept RMAN has its own command language, separate from the operating system.

Command Types

Stand-alone Commands

Executed individually at the RMAN prompt

BACKUP DATABASE;

Type the command with a semicolon (;) and press Enter. RMAN executes it immediately.

Job Commands (RUN Blocks)

Grouped together and executed sequentially

RUN {
   command1;
   command2;
   command3;
}

If any command fails, RMAN stops executing remaining commands. Effects of executed commands are not undone.

RMAN Persistent Settings

RMAN behavior is controlled by configuration settings stored in the control file and optionally in the recovery catalog.

View all settings:
SHOW ALL;
View specific setting:
SHOW <setting>;
Query via SQL:
SELECT * FROM V$RMAN_CONFIGURATION;

Modes of Issuing RMAN Commands

Mode Description Best For
Interactive Type commands directly at the prompt One-time operations, reports, specific backups/recoveries
Batch Commands written in a file, run using rman @script_file Automated jobs
Pipe Interface Commands via DBMS_PIPE PL/SQL package External applications (rarely used by DBAs)

Backup Strategy

Backup Options

Whole Database Backup

  • Includes all data files
  • Includes control file
  • Backs up entire database

Partial Database Backup

  • Specific tablespaces only
  • Specific data files only
  • May include/exclude control file

Backup Types

Type Description Requirements
Full Backup Backs up all used data blocks in specified files -
Incremental Backup Backs up only blocks changed since previous backup Requires Level 0 backup first
Important Note Full backup ≠ Whole backup
  • Full backup: backs up all used blocks
  • Whole backup: backs up the entire database
  • Both can be combined (e.g., full backup of the entire database)

RMAN BACKUP Command

Backup Types by Consistency

🔥 Inconsistent Backup (Hot)

  • Taken while database is open
  • SCN numbers may not match control file
  • Can only be done in ARCHIVELOG mode
  • Recovery requires applying archived redo logs

❄️ Consistent Backup (Cold)

  • Taken while database is mounted after clean shutdown
  • SCN and checkpoint timestamps synchronized
  • Can be done in ARCHIVELOG or NOARCHIVELOG mode
  • In NOARCHIVELOG mode, changes after backup are lost if failure occurs

Performing Backups

Online Backup (Open Database):
BACKUP DATABASE PLUS ARCHIVELOG;

Backs up data files and archived redo logs. RMAN takes two backups of archived redo logs (before and after data file backup) to ensure recoverability.

Offline Backup (Mounted Database):
BACKUP DATABASE;

Requires shutdown and mount before backup. Database can be started after backup completes.

Partial Backups

Backup specific tablespaces:
BACKUP TABLESPACE users, tools;
Backup data files by number:
BACKUP DATAFILE 1,2,3,4;
Backup data files by full path:
BACKUP DATAFILE '/u01/oradata/orcl/users01.dbf';

Backup File Locations and Naming

  • Default storage: FRA (Flash Recovery Area)
  • FORMAT option: Specify path and file name pattern
Custom format:
BACKUP DATABASE FORMAT '/u02/orcl_%U';

%U generates a unique string to ensure unique file names

ASM diskgroups:
BACKUP DATABASE FORMAT '+DGNAME';

ASM automatically generates unique names

Backing Up Control File and SPFILE

Control File Backup Methods

1️⃣ Automatic Backup (Autobackup)

CONFIGURE CONTROLFILE AUTOBACKUP ON;

ON by default from 12.2 onward

Occurs after every RMAN backup automatically

2️⃣ Manual Backups

BACKUP DATABASE INCLUDE CURRENT CONTROLFILE;
BACKUP CURRENT CONTROLFILE;
BACKUP AS COPY CURRENT CONTROLFILE;

3️⃣ Indirect Method

Backing up data file #1 automatically includes the control file and SPFILE

SPFILE Backup

  • SPFILE contains initialization parameters for the database
  • Autobackup of control file includes a backup of SPFILE automatically
Manual backup command:
BACKUP SPFILE;

Trace File Backup

Generates SQL statements to recreate control file if all backups are lost

ALTER DATABASE BACKUP CONTROLFILE TO TRACE;

Saved under $DIAGNOSTIC_DEST directory

Note: Cannot be executed directly; must follow the instructions in the trace file.

Archivelog Backup Commands

BACKUP ARCHIVELOG Command

All available archivelogs:
BACKUP ARCHIVELOG ALL;
Delete after backup:
BACKUP ARCHIVELOG ALL DELETE INPUT;
Backup by SCN range:
BACKUP ARCHIVELOG FROM SCN 1000 UNTIL SCN 2000;
Backup by sequence number:
BACKUP ARCHIVELOG SEQUENCE 100 UNTIL 200;
Backup by time:
BACKUP ARCHIVELOG FROM TIME "SYSDATE-1";

BACKUP DATABASE PLUS ARCHIVELOG

Simplifies backup process by including archived redo logs automatically

Internal Steps RMAN Performs:
  1. Runs ALTER SYSTEM ARCHIVE LOG CURRENT
  2. Runs BACKUP ARCHIVELOG ALL
  3. Backs up the specified objects
  4. Runs ALTER SYSTEM ARCHIVE LOG CURRENT again
  5. Backs up any remaining archived redo logs
BACKUP DATABASE PLUS ARCHIVELOG;
Recommendation: Use BACKUP DATABASE PLUS ARCHIVELOG for online backups to ensure recoverability.

Incremental Backups

Overview

Only backs up changed blocks since the last backup, not the entire database.

⚡ Benefits

  • Faster than full backups
  • Less storage space required
  • Lower I/O bandwidth needed
  • Can back up NOLOGGING objects

Level 0 and Level 1 Backups

Level Description File Types
Level 0 Full backup of all database blocks. Forms the base for all Level 1 backups. Backupsets or image copies
Level 1 Backs up only changed blocks Backupsets only (not image copies)

Differential vs Cumulative

Feature Differential Cumulative
What it backs up Changes since last incremental backup Changes since last Level 0 backup
Backup size Smaller Larger
Recovery Reads all Level 1 backups + Level 0 Reads only most recent Level 1 + Level 0
Advantage Saves storage space Faster recovery
Recommendation: Differential backups are preferred in large databases to save disk space.

Incremental Backup Commands

Default (Differential):
BACKUP INCREMENTAL LEVEL 1 DATABASE;
Cumulative:
BACKUP INCREMENTAL LEVEL 1 CUMULATIVE DATABASE;

Block Change Tracking (BCT)

Speeds up incremental backups by tracking changed blocks in a tracking file.

Enable BCT:
ALTER DATABASE ENABLE BLOCK CHANGE TRACKING USING FILE '/u01/oradata/bct_file.bct';
Disable BCT:
ALTER DATABASE DISABLE BLOCK CHANGE TRACKING;
BCT Key Points:
  • Can track up to 8 incremental backups
  • After 7 incremental backups, take a full Level 0 backup
  • BCT file grows in 10 MB increments
  • Not suitable if >20% of database changes frequently
  • Not enabled by default
  • RMAN does not backup or restore BCT file

Best Practices for Incremental Backups

📅 Weekends

Take Level 0 incremental backup

📆 Weekdays

Take Level 1 incremental backups (off-peak hours)

💾 Backup Type

Prefer differential incremental backups over cumulative

⚡ Performance

Enable Block Change Tracking for large databases

Backup Retention Policy

Retention policy determines which backups RMAN keeps or marks as obsolete.

Types of Retention Policies

📊 Redundancy-Based

Keeps a fixed number of backups

CONFIGURE RETENTION POLICY TO REDUNDANCY 3;

Keeps last 3 backups, older ones marked obsolete

⏰ Window-Based (Time-Based)

Keeps backups for a specified period

CONFIGURE RETENTION POLICY TO RECOVERY WINDOW OF 7 DAYS;

Allows recovery to any point in last 7 days

Important Notes:
  • Only one retention policy can be active at a time
  • Default redundancy is 1 backup
  • Always configure a retention policy to avoid storage issues in FRA

Disabling Retention Policy

CONFIGURE RETENTION POLICY TO NONE;
Not Recommended: RMAN won't consider any backup obsolete. Especially problematic when using FRA.

RMAN Recovery Overview

Types of Database Failures

🔧 Physical Failures

Problems with actual database files (lost or corrupted)

💭 Logical Failures

Issues caused by applications or users (accidental or intentional data changes)

RMAN Recovery Concepts

Command Purpose Description
RESTORE Creates database file from backup Restores file to backup point, not the failure time
RECOVER Applies changes Uses archived redo logs, incremental backups, and online redo logs
Recovery Types:
  • Complete Recovery: Up to failure point
  • Incomplete / Point-in-Time Recovery (PITR): Up to a specified time before failure

Critical vs Non-Critical Files

❌ Non-Critical Files

Can be lost without crashing the DB

  • Multiplexed redo logs
  • Index tablespaces
  • Init parameter files
  • Inactive temp files

🚨 Critical Files

Loss causes database crash

  • Control files
  • System tablespace datafiles

Whole Database Recovery

NOARCHIVELOG Mode

Cannot recover to the point of failure unless incremental backups exist.
Recovery Steps:
-- Start database in MOUNT state STARTUP MOUNT; -- Restore database RESTORE DATABASE; -- Recover database (typically does nothing in NOARCHIVELOG mode) RECOVER DATABASE; -- Open with RESETLOGS ALTER DATABASE OPEN RESETLOGS;

ARCHIVELOG Mode

Can perform complete or incomplete recovery. Most production databases use ARCHIVELOG mode.
Recovery Steps:
STARTUP MOUNT; RESTORE DATABASE; RECOVER DATABASE; ALTER DATABASE OPEN;

Recovery to a New Location

RUN { SET NEWNAME FOR DATAFILE 1 TO '/new/location/system01.dbf'; SET NEWNAME FOR DATAFILE 2 TO '/new/location/sysaux01.dbf'; RESTORE DATABASE; SWITCH DATAFILE ALL; RECOVER DATABASE; }

User Tablespace Recovery

Only affects the specific tablespace, not SYSTEM, SYSAUX, UNDO, or TEMP tablespaces.

Recovery Steps:
-- Take tablespace OFFLINE ALTER TABLESPACE users OFFLINE; -- Restore tablespace RESTORE TABLESPACE users; -- Recover tablespace RECOVER TABLESPACE users; -- Bring tablespace ONLINE ALTER TABLESPACE users ONLINE;

Recovery by Switching Data Files

Instead of restoring datafiles from backup sets, switch the database to use existing data file image copies.

Advantage: Faster than restoring from backups because RMAN updates the control file instead of creating new files.

Scenario: All Database Datafiles Lost

-- Mount the database STARTUP MOUNT; -- Switch to data file copies SWITCH DATABASE TO COPY; -- Recover database RECOVER DATABASE; -- Open database ALTER DATABASE OPEN;

Scenario: Single User Tablespace Datafile Lost

-- Take datafile offline ALTER DATABASE DATAFILE 5 OFFLINE; -- Switch to image copy SWITCH DATAFILE 5 TO COPY; -- Recover datafile RECOVER DATAFILE 5; -- Bring datafile online ALTER DATABASE DATAFILE 5 ONLINE;

Point-in-Time Recovery (PITR)

Restore the database or parts of it to a previous state (e.g., accidental data changes or missing archive logs).

Levels of PITR

🗄️ Database Level

DBPITR - Full database recovery

📦 Tablespace Level

TSPITR - Single tablespace recovery

📋 Table Level

Recover specific tables

Recovery Point Options

  • SCN (System Change Number)
  • Redo log sequence number
  • Restore point
  • Timestamp

Oracle converts all to SCN internally

DBPITR Requirements

  • Database must be in ARCHIVELOG mode
  • Backups of all datafiles must exist before the required recovery SCN

DBPITR Procedure

Step-by-Step Recovery:
-- Shutdown database SHUTDOWN IMMEDIATE; -- Start RMAN and run recovery RUN { SET UNTIL TIME 'yyyy-mm-dd hh24:mi:ss'; RESTORE DATABASE; RECOVER DATABASE; } -- Open with RESETLOGS ALTER DATABASE OPEN RESETLOGS;
RESETLOGS Required: After DBPITR, the database must be opened with RESETLOGS, which:
  • Archives online redo logs if present
  • Clears or recreates online redo logs
  • Resets log sequence to 1
  • Updates all files with a new RESETLOGS SCN
  • Creates a new database incarnation

Best Practice: Always use TAGs in production environments for better backup management.
