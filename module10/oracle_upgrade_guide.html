Oracle Database Upgrade Guide

A complete reference covering upgrade methods, the AutoUpgrade utility, pre/post-upgrade actions, and Grid Infrastructure upgrades.

Module 1
Upgrade Methods
Direct Upgrade
1
Install new Oracle release in a separate home on the same machine
2
Use an upgrade tool to update data dictionary objects
3
Database container re-linked to the new home
4
Old home link no longer exists post-upgrade
Data Migration
1
Install new Oracle release (same or different machine)
2
Create a brand new database from the new home
3
Move schemas using Data Pump or similar tools
4
Application connects to the new database
⚠️
Data Migration is not technically an upgrade. This guide focuses on the Direct Upgrade method. Supported source releases: 11.2.0.4+ for 19c target; 20cR2+ for 21c target.
💡
Upgrading from 12cR1 → 21c? You must first upgrade to 19c, then to 21c. Multi-hop upgrades are required for unsupported source versions.

Pre-Upgrade Checklist  — click items to mark done
System & Software
Verify OS version meets new release requirements
Check memory and disk space requirements
Verify oracle user is in the oper group
Verify REMOTE_LOGIN_PASSWORDFILE ≠ SHARED
Create / migrate the database password file
Verify client/server compatibility (Doc 207303.1)
Database Preparation
Take a full backup of the database
Empty the recycle bin in all containers
Gather statistics on data dictionary & fixed tables
Install new Oracle Home (follow naming convention)
Copy network config files to new TNS_ADMIN location
Stop old listener; create new listener from new home
Notify application users of expected downtime
AutoUpgrade Preparation
Download latest autoupgrade.jar — MOS Doc 2485457.1
Copy jar to new Oracle Home directory
Create a log directory for AutoUpgrade output
Create the AutoUpgrade configuration file
Run AutoUpgrade in Analyze mode; review HTML report
Implement recommended fixes from the Analyze report
Re-run Analyze to confirm database is ready
Module 3
Oracle Upgrade Tools
🌟
AutoUpgrade 19c+ RECOMMENDED
The primary recommended tool. Detects issues, provides fixup scripts, supports multiple simultaneous upgrades, handles Standalone / ASM / RAC, supports non-CDB → PDB conversion. Single jar file, updated via MOS.
⚡
Parallel Upgrade Utility 12c+
Also known as catctl.pl / dbupgrade. Reduces upgrade time by loading the dictionary in parallel using multiple SQL processes.
🚢
Fleet Patching & Provisioning (FPP) TOOL
Previously "Rapid Home Provisioning". Software lifecycle management for patching and upgrading thousands of databases in parallel across a fleet.
🖥️
Fleet Maintenance (OEM Plugin) LICENSE REQUIRED
Oracle Enterprise Manager plugin. Requires Database Lifecycle Management Pack license. GUI-based group patching and upgrading.
🪦
DBUA (Database Upgrade Assistant) AVOID
GUI-based, still supported in 19c but not recommended. Designed for Oracle 11gR2 and earlier. AutoUpgrade supersedes it entirely.
Module 3
AutoUpgrade — Processing Modes  — click a mode to expand
🔍
ANALYZE
Checks DB for issues. Generates report. No changes made.
🔧
FIXUP
Examines DB and applies some or all recommended fixes.
🚀
DEPLOY
Full upgrade procedure — all stages end to end.
⬆️
UPGRADE
Upgrade + post-fixups on a DB already in the new home.
ANALYZE — Run this first. Scans the source database for compatibility issues and generates an HTML report (errors, warnings, recommendations). Creates fixup scripts but does not execute them.
# Run from the NEW Oracle Home
$NEW_ORACLE_HOME/jdk/bin/java -jar autoupgrade.jar \
  -config upgrade.cfg -mode analyze
FIXUP — Similar to Analyze, but also executes the fixup scripts. Most useful when upgrading multiple similar databases — verify scripts on one, then apply to the rest automatically.
$NEW_ORACLE_HOME/jdk/bin/java -jar autoupgrade.jar \
  -config upgrade.cfg -mode fixup
DEPLOY — Complete upgrade procedure. Runs all AutoUpgrade stages (Analyze stages are re-executed). The database must be shut down and applications disconnected before starting.
$NEW_ORACLE_HOME/jdk/bin/java -jar autoupgrade.jar \
  -config upgrade.cfg -mode deploy
UPGRADE — Runs upgrade + post-fixups on a database already moved to a new Oracle Home (e.g., after cross-system migration). Useful when database files were manually relocated to a different server.
$NEW_ORACLE_HOME/jdk/bin/java -jar autoupgrade.jar \
  -config upgrade.cfg -mode upgrade
💡
Typical workflow: ANALYZE → review report → fix issues → DEPLOY. Use FIXUP as an optional automated-fix step when upgrading similar database fleets.
Module 3
AutoUpgrade — Configuration File
Parameter Types
G
Global — apply to all upgrade jobs.
global.<param> = <value>
L
Local — apply to a specific upgrade job.
<config_name>.<param> = <value>
!
Local parameters override global ones for that job
Minimum Required Parameters
1
global.autoupg_log_dir — log files directory
2
[job].source_home — source Oracle Home path
3
[job].target_home — target Oracle Home path
4
[job].sid — database instance name
Example — Complete Minimal Config
# Global parameter
global.autoupg_log_dir=/u01/autoupg_logs

# Local parameters for job "19c21c"
19c21c.source_home=/u01/app/oracle/product/19.0.0/dbhome_1
19c21c.target_home=/u01/app/oracle/product/21.0.0/dbhome_1
19c21c.sid=CDB1

Module 4
Post-Upgrade Checklist  — click items to mark done
Immediate Actions
Drop the Guaranteed Restore Point (GRP) from AutoUpgrade
Update ORACLE_HOME in .bash_profile to new home
Source bash_profile; verify sqlplus runs from new home
Verify database is open for READ WRITE
Verify all DB services registered in the new listener
Test application connectivity through listener (tnsnames)
Validation & Cleanup
Compile invalid PL/SQL objects (use catcon.pl if many)
Run opatch lsinventory from new home — must succeed
Upgrade tables dependent on Oracle maintained types (12cR2+)
Update COMPATIBLE parameter after stability confirmed
Restart the instance to apply COMPATIBLE change
Deinstall old Oracle Home (after a few days, if no rollback needed)
COMPATIBLE Parameter — Best Practice
1
After upgrade, COMPATIBLE still reflects the old release number — intentional
2
New release features remain disabled until the parameter is updated
3
Allow users to work for a few days — confirm stability first
4
Update COMPATIBLE to new version number, then restart the instance
5
If an app needs a new feature immediately, updating right away is acceptable
💡
In production: also take a full backup post-upgrade and apply the most recent Release Update (RU) patch to the new Oracle Home.
Module 5
Grid Infrastructure Upgrade
⚠️
AutoUpgrade does NOT upgrade Grid Infrastructure. When Oracle DB runs with GI (Oracle Restart, RAC, RAC One Node), upgrade GI first using the Oracle Universal Installer, then upgrade the database with AutoUpgrade.
GI Upgrade Steps
Download GI 21c software (Linux x86-64)
Take a VM snapshot before starting
Create new grid home directory (as grid user)
Extract installation zip into new grid home
Run runcluvfy (CVU) to verify prerequisites
Shut down the database (not other grid resources)
Unset grid environment variables
Run installer → select "Upgrade Oracle Grid Infrastructure"
Provide root password and complete installation
GI Post-Upgrade Actions
Verify Clusterware resources are online
Update ORACLE_HOME in .bash_profile for grid user
Source bash_profile and verify variables are correct
Detach old GI home from Oracle inventory
Verify old home is marked as REMOVED
Upgrade COMPATIBLE attribute of disk groups
Start the database and confirm normal operation
💡
The database will technically run on the old home after GI upgrade. However, Oracle recommends matching release and patch levels for both GI and the database. Proceed to upgrade the DB with AutoUpgrade afterward.
