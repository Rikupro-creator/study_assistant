Part 01

Introduction & Tuning Areas

The Three Areas of Oracle Performance Tuning

Oracle performance tuning spans three intersecting areas. Performance Tuning Tools are the instruments DBAs reach for — V$ views, OS utilities, database-callable facilities, and third-party products. SQL Tuning involves writing efficient SQL, influencing the optimizer, and creating structures like indexes that help it. Instance Tuning addresses the database at the structural level: the SGA, disk I/O, background processes, client sessions, and the OS layer beneath it all.

Scope This course focuses on Performance Tuning Tools and Instance Tuning. SQL Tuning is not covered.

Performance Issue Attributes

Before diagnosing any issue, establish what you're dealing with by answering five questions:

Scope

Is the entire system affected, or just part of it? A full-system slowdown points to shared resources (CPU, I/O subsystem). A partial issue — one report taking 2 minutes, an ETL that grew from 3 hours to 12 — points somewhere specific.

Slow or Hanging?

A hanging situation — where the system appears frozen — is also a performance issue. The diagnosis path differs from a slow-response investigation.

Permanent or Intermittent?

Intermittent issues are harder to catch and diagnose. They require longer observation windows and often more invasive instrumentation.

Reproducible?

Reproducible issues are dramatically easier to trace — you can instrument and observe. Non-reproducible ones require relying on historical data.

Errors Involved?

Some errors are performance issues in disguise. A full archivelog destination hangs the database. An undersized shared pool triggers ORA-04031. Always check the alert log early.

Oracle's Performance Tuning Methodology

Because tuning touches every layer of the IT stack, there is no universal procedure — only a methodology. Two DBAs may follow different steps yet apply the same approach. This methodology begins after gathering issue attributes.

1
Set a measurable target

Ask how long the operation used to take. That baseline becomes your target. Without a target, you have no way to know when you're done.

2
Gather technical symptoms

The most important step — everything downstream depends on it. Use AWR reports, comparison AWR reports, V$ views, ADDM, the alert log, SQL tracing, and OS utilities. "Symptoms" here means technical indicators, not user descriptions.

3
List possible causes

Map the symptoms to known causes. For unfamiliar symptoms, consult Oracle Support documentation.

4
Develop a trial solution

Could be as simple as creating an index or following an ADDM recommendation. Could involve configuration or structural changes.

5
Test in a test environment

Mirror production as closely as possible. Use the same workload — Oracle's Database Replay tool can capture and replay production workloads on a test system.

6
Check the outcome

If the target is hit, stop. Further optimization beyond the target is a proactive development task, not tuning. If the target is missed, return to Step 3 with a different approach.

Emergency Exception When the entire system is unavailable — archivelog destination full, for instance — treat it as an emergency. Skip test environment validation and act immediately to restore service.

Describing System Workload

A DBA should be able to quantify workload in concrete terms for both tuning and stakeholder communication. Example: 100 users, one transaction each every 5 minutes, over an 8-hour day → 20 transactions/minute on average, 9,600/day. But workload isn't flat — always design for peak, not average. In this case, a reasonable design target might be 120 transactions/minute.

Part 02

IT Systems & Hardware

Server Performance Components

Oracle runs on top of an OS, which runs on top of hardware. Understanding the hardware layer is essential context for database tuning. The four components that govern server performance are CPU, memory, storage, and network. In production, you cannot change hardware specs — you monitor utilization and rate of consumption against the fixed capacity.

CPU

Performance is shaped by socket count, core count, cache size and level, clock speed, and processor architecture. More cores means more concurrent work can be handled — and typically higher Oracle licensing cost.

Memory

Type (DDR4, RDIMM, LRDIMM, NVDIMM) and transfer rate (MT/s) determine raw speed. The critical concern for a DBA: if the OS cannot find free memory, it swaps to disk — orders of magnitude slower. Monitor free memory closely.

Storage

Mechanical disks are a historic bottleneck; SSDs are far faster. Storage performance is measured in IOPS, latency, and throughput (MB/s). RAID configuration and connection type also affect effective performance. For multiple storage arrays, watch load distribution — an imbalanced setup creates a hidden bottleneck on the most-used path.

Network

Measured in bandwidth — 1 Gbps and 10 Gbps Ethernet are common. Utilization approaching 90% causes packet queuing. Persistent faulty packet reports signal a network health issue requiring investigation.

The IT Technology Stack

A typical system runs: database at the back, accessed by an application server via network, with clients at the front. Real deployments add firewalls, load balancers, and internet connections. A problem at any layer can surface as a performance issue at any other.

The most impactful layer for performance is application logic. Developers historically optimize for correctness, not efficiency. The classic example: a full table scan returning a few rows causes massive I/O contention for even a handful of users. A single well-chosen index transforms that into a few block reads. Most Oracle tuning targets exactly this — minimizing resource consumption per operation.

Course Scope This course focuses on the database layer. The goal is to identify whether a performance issue originates there, and if poorly written SQL is involved, isolate those statements for tuning or developer escalation.
Part 03

Performance Tuning Tools & Statistics

Oracle's Performance Tuning Toolset

V$ performance views live in memory and reset on instance restart — the most commonly used DBA diagnostic tool. Data dictionary views persist across restarts and live in tablespaces. AWR (Automatic Workload Repository) is the primary snapshot and reporting tool, but requires a separate license. Without it, use Statspack. ADDM proactively analyzes AWR data to identify and predict issues. Advisors provide targeted recommendations for specific areas. The alert log and trace files (auto-generated or manually triggered) round out the toolset. GUI access is available via EM Database Express (free) or OEM Cloud Control (licensed).

The Three Types of Performance Statistics

Type 01
Cumulative

Raw counts accumulated since the last instance startup. Reset on restart. Most are only meaningful compared over time. Some are intrinsically useful — enqueue waits should barely appear in a healthy system. Examples: time model, instance wait events.

Type 02
Metrics

The rate of change of a cumulative statistic, measured against time, transactions, or calls. The primary tool for detecting issues before they escalate. Power ADDM, AWR, and server alerts. E.g., database calls per second.

Type 03
Sampled

Statistics saved as periodic snapshots, enabling historical look-back. Lets you diagnose issues that occurred in the past. Main sources: ADDM, AWR, and Active Session History (ASH).

STATISTICS_LEVEL Parameter

Controls the depth of statistics Oracle collects. Three values:

BASIC
Disables most performance statistics and all automatic advisors. Use only for static-data, static-load databases where continuous collection adds no value.
TYPICAL
Default. Collects all major statistics, runs advisors on schedule. Sufficient for most production systems.
ALL
Adds timed OS statistics and execution plan statistics. Higher overhead. Used during active performance investigations. Can be set at session level — no need to change the whole instance when tuning one session.

To see which internal parameters each setting affects, query V$STATISTICS_LEVEL.

Part 04

The Time Model

DB Time, DB CPU & DB Wait Time

When no client request is being served, the database registers idle wait events. When a request arrives, the database consumes CPU, memory, and storage to process it. The time spent actively executing is DB CPU. Because resources are shared, the database often waits for a busy resource before proceeding — each such wait is a named wait event. The total of all non-idle waits is DB Wait Time.

These two together make DB Time — the total time the database spends actively serving client requests:

Formula DB Time = DB CPU + Total Non-Idle Wait Time

The Time Model is the set of statistics that classifies where DB time was spent. In Oracle 12c, it has roughly 20 categories — SQL execution elapsed time, hard/soft parse time, PL/SQL execution elapsed time, and others.

Important: DB Time Can Exceed Clock Time

DB time aggregates across all sessions. With 8 CPU cores, one real-time minute provides 8 CPU-minutes of capacity. If 10 sessions are each waiting simultaneously, all their wait times stack into DB time. A DB time of 2 hours in one real-time hour is perfectly normal on a busy, multi-core system.

Time Model Views

V$SYS_TIME_MODEL — system-wide accumulated times since last startup, in microseconds, organized hierarchically with DB time at the top. A second tree alongside it tracks Background Elapsed Time (excluded from DB time, but included to capture RMAN's impact since it's DBA-initiated and resource-intensive).

V$SESS_TIME_MODEL — same structure at the individual session level. Use it to identify which sessions consume the most DB time or DB CPU. Note: terminated sessions are not visible.

Why DB Time Matters

DB time is the single best indicator of database workload. It rises with load and falls when the database is idle. Beyond that, it enables four key diagnostic uses:

1
Measure overall load

If DB time is growing, the database is receiving more work. If it's flat, workload is stable.

2
Assess wait ratio

If total wait time is 50%+ of DB time, the system is spending more time waiting than working. This is a red flag for a poorly performing system.

3
Measure change impact

Added memory? Tuned a query? DB time before and after quantifies the real impact of the change.

4
Spot operation anomalies

If hard parses represent an unusually high share of DB time, that's worth investigating further — it's not normal in a well-tuned system.

Measuring Scalability with DB Time

As user load increases, both DB CPU and wait time should rise in proportion — that's a scalable system. At some point, wait time starts growing much faster than CPU time. That inflection is where users begin noticing degradation. Example data point: at 10 users, waits = 18% of DB time. At 50 users, waits = 47% (wait time +900s, CPU +160s). At 100 users, waits = 70% — clearly unacceptable.

Historical Data: DBA_HIST_SYS_TIME_MODEL

V$SYS_TIME_MODEL accumulates since startup and can't tell you what happened in a specific window. Oracle snapshots it hourly into DBA_HIST_SYS_TIME_MODEL. While queryable directly, the easiest way to read it is through an AWR report, which consolidates most DBA_HIST views into one readable document.

Part 05

Instance Activity & Wait Events

Instance Activity Views

While the time model gives high-level DB time breakdowns, instance activity views provide granular counters for hundreds of specific operations. The primary view is V$SYSSTAT — instance-level statistics accumulated since startup. It holds a statistic name and its cumulative value. Statistics are identified by name (not number — numbers can change across Oracle versions, names don't). Commonly watched: parse time CPU, physical reads, user commits.

To explore what's available with descriptions, query V$STATNAME. For service-level stats: V$SERVICESTATS. For session-level: V$SESSTAT (join to V$STATNAME for names). For the current session specifically: V$MYSTAT. For object-level detail including wait events broken down by segment: V$SEGMENT_STATISTICS — this one includes statistic names natively, so no join needed.

Wait Events

A wait event is a named record of the database waiting for a specific resource. There are over 600. The real-time diagnostic views are V$SESSION_WAIT and V$SESSION, which show events currently held by active sessions.

Reading Wait Columns

STATE
WAITING — currently waiting. WAITED UNKNOWN TIME — just finished, duration unknown (set TIMED_STATISTICS=TRUE). WAITED SHORT TIME — under 1/100s. WAITED KNOWN TIME — finished, duration in WAIT_TIME.
WAIT_TIME
Positive = last wait duration in 1/100s. 0 = still waiting. -1 = under 1/100s. -2 = timing unavailable.
SECONDS_IN_WAIT
Current wait duration in seconds. Only meaningful when STATE = WAITING.
P1, P2, P3
Event-specific parameters (object, file, block, or lock ID). Names in P1TEXT, P2TEXT, P3TEXT. Documented in Oracle Reference Appendix C.
Quick Rule Session is WAITING? → look at SECONDS_IN_WAIT. Wait has finished? → look at WAIT_TIME.

Common Wait Events

Buffer busy waits — process waiting for a data block in the buffer cache to become available. Free buffer waits — buffer cache is full. DB file sequential read — single-block read (index lookup, etc.). DB file scattered read — multi-block read (full table scan, etc.). Enqueue waits — serializing concurrent access to a resource. Library cache waits — pinning/locking an object in the library cache. Log buffer space — redo log buffer full. Log file sync — waiting for redo write to complete.

Key Principle Wait events are symptoms, not causes. They point toward the root cause but are not the root cause themselves. Some belong at the top of the wait list in a healthy system. Others should never appear there — their presence signals a real problem. Learning which is which is a major part of this course.
Part 06

Metrics & Event Histograms

What Metrics Tell You That Statistics Don't

A cumulative statistic just accumulates upward — it shows total volume but not pace or timing. A metric shows the rate of change of that statistic over a short interval. A graph of accumulated physical reads looks like a ramp. The same data as a metric shows exactly when reads spiked and by how much. That's the diagnostic value.

Most metrics are time-based (per second), but some use other dimensions (per transaction, per session). Metrics feed ADDM, AWR, and server alert systems. The full catalog is in V$METRIC_NAME. Oracle computes metrics for system, session, file, and wait event statistics.

Metric Views

V$-prefixed metric views live in memory, covering the last hour, updated every 15–60 seconds (varies by metric). When an AWR snapshot is taken, these are copied to their DBA_ counterparts with a SNAP_ID added. V$SYSMETRIC_HISTORY → DBA_HIST_SYSMETRIC_HISTORY. Not all V$ metrics are copied to DBA_ views. In normal operations, you rarely query these directly — tools consume them for you — but you may reference them to verify tool outputs.

Event Histograms

A metric spike (say, physical reads per second) could mean normal load growth or a system problem. Histograms let you distinguish the two. They bucket wait event occurrences by duration — each bucket represents a time range, and its value is how many waits fell there. Buckets double: <1ms, 1–2ms, 2–4ms, 4–8ms, 8–16ms, 16–32ms, 32–64ms…

If your storage normally delivers I/O in ~10ms and the histogram shows most waits concentrated in the 32–64ms bucket — something is wrong with the storage subsystem. If most waits sit in the 8–16ms bucket — that's close to normal, and the spike is likely just higher user load or a batch job starting.

Histogram Views

V$EVENT_HISTOGRAM — ~148 events, in milliseconds. V$EVENT_HISTOGRAM_MICRO — same data in microseconds. DBA_HIST_EVENT_HISTOGRAM — AWR-backed historical version. V$FILE_HISTOGRAM — single-block reads per data file.

Practical Note AWR reports present histogram data in tabular format, which is easier to read than querying V$ views directly. Know how to interpret histograms; you usually won't need to query them manually.
Don't Confuse These are wait event histograms. Optimizer statistics histograms are a completely separate concept used in SQL tuning — not relevant here.
Part 07

Server Generated Alerts

Proactive Alerting

Oracle's server generated alerts fire automatically when a metric crosses a predefined threshold, or when a specific event occurs. Threshold-based examples: physical reads/sec, user commits/sec, SQL service response time. Event-based examples: snapshot too old, resumable session suspended, recovery area space usage.

Two severity levels exist. A warning fires at the lower threshold — early notice. A critical fires at the upper threshold — needs immediate attention. Manage alerts via OEM (easiest) or programmatically via the DBMS_SERVER_ALERT package.

Setting Thresholds: DBMS_SERVER_ALERT.SET_THRESHOLD

METRIC_ID
The metric to monitor. Full list in PL/SQL Packages Reference, Table 143-3.
WARNING_OPERATOR
Comparison operator for warning threshold. Uses package constants for =, >, >=, <, <=, !=, list, or DO_NOT_CHECK (disables warning).
WARNING_VALUE
The warning threshold value. NULL = no warning alert.
CRITICAL_OPERATOR / CRITICAL_VALUE
Same pattern, for the critical level.
OBSERVATION_PERIOD
Minutes over which metric is computed and checked. Default: 1 minute.
CONSECUTIVE_OCCURRENCES
How many consecutive violations before the alert fires. Default: 1. Increase to tolerate expected brief spikes — e.g., short session blocking in a heavy-load app.
OBJECT_TYPE / OBJECT_NAME
Scope the threshold to a specific object type (system, file, service, tablespace, session) and named object. Constants in Table 143-1 of the documentation.

Default Thresholds & Retrieving Alert Info

Out of the box, only one threshold is active: tablespace space usage — warning at ≥85%, critical at >97%. No performance metrics are monitored by default. Query DBA_THRESHOLDS to see configured thresholds. Use DBMS_SERVER_ALERT.GET_THRESHOLD to retrieve a specific metric's threshold programmatically.

For generated alerts: DBA_OUTSTANDING_ALERTS (active, uncleared), DBA_ALERT_HISTORY (cleared), DBA_ALERT_HISTORY_DETAIL (both).

Best Practices

No universal thresholds exist — every system behaves differently. Build your baseline by reading AWR reports repeatedly while the system is healthy, then configure thresholds that flag meaningful deviations. For critical systems, connect alerts to email or SMS. If you have OEM, use its alerting system instead of the PL/SQL package.

Batch Processing Trap Thresholds set for normal OLTP workload will fire false alarms during batch windows. Oracle recommends adaptive thresholds for metrics like this — they learn from system behavior patterns and adjust automatically. Covered in a later lecture.
Remember Server alerts supplement your monitoring — they don't replace it. OS-level metrics and statistics outside Oracle's standard alert framework still need separate monitoring.
Part 08

ADDM — Automatic Database Diagnostic Monitor

What ADDM Does

ADDM reads AWR snapshots for a target period — and in-memory views where needed — to detect existing performance issues and predict ones likely to emerge soon. It then produces findings and recommendations. By default it runs automatically after each AWR snapshot, analyzing the interval between the two most recent. It can also be invoked manually across any snapshot range. All results are stored in the AWR.

ADDM's performance indicator is DB time. Every finding and recommendation aims at reducing DB time for the detected bottleneck. ADDM performs analysis in two dimensions simultaneously: user-request operations (connection time, SQL execution, user calls) and resource metrics (CPU, memory, I/O, database locks). In both dimensions it does cause-and-effect analysis — symptoms first, then root cause.

License Required ADDM requires the same license as AWR. No AWR license = no ADDM.

Accessing ADDM

Via EM Express

Performance Hub → ADDM tab. Lists tasks by time period. Each task shows finding count and average active sessions. Click a task to see actionable findings and informational findings. The informational panel tells you what's not a problem — useful for narrowing scope. Each actionable finding shows its DB time impact percentage, recommendations, rationale, and expected benefit.

In the AWR Report

ADDM appears as the first section after the header, and again as a full copy at the report's end. Findings are ordered by percentage impact on average active sessions. Average active sessions = DB time ÷ elapsed time (both in the report header).

When reading: check dependencies before acting on order. If findings include top SQL, high CPU usage, and undersized SGA — investigate CPU and SGA first. A slow SQL statement may simply be a symptom of the SGA being too small. Fix the root cause, generate a fresh AWR report, and confirm the downstream findings have cleared.

From SQL*Plus

Use DBMS_ADVISOR.GET_TASK_REPORT(task_name). Task names come from data dictionary views. Output is text only — no HTML — but well structured and readable.

Perspective ADDM is a tool, not a performance expert. It identifies areas of concern. Validating its recommendations and understanding their dependencies is your job as a DBA.

Running ADDM Manually

Required when the default two-snapshot window is too narrow. Condition: no instance restart within the analysis period.

A
DBMS_ADVISOR package

Create a task → set begin/end snapshot IDs via SET_TASK_PARAMETER → execute with EXECUTE_TASK → retrieve with GET_TASK_REPORT.

B
Run addmrpt.sql (simpler)

Prompts for begin/end snapshot IDs and a report filename. Generates and saves automatically. Use addmrpti.sql for RAC.

Part 09

ADDM Comparison Report

Purpose

The ADDM comparison report measures the performance impact of a specific change — a parameter tweak, index creation, hardware upgrade, or any planned intervention. Unlike the AWR comparison report (which surfaces statistical deltas for the DBA to interpret), the ADDM comparison report attempts the interpretation for you, linking detected system changes to observed performance differences via cause-and-effect analysis.

Base Period vs Compare Period

The base period covers AWR snapshots from before the change — the normal system state. The compare period covers snapshots after the change, whether that represents a regression or an improvement. ADDM identifies configuration changes (SGA size, parameters), measures the performance delta, and tries to map changes to outcomes.

SQL Commonality

SQL commonality measures how similar the SQL workload is between the two periods (0–100%). The higher, the more reliable the comparison. Treat findings as reliable only at 80%+ commonality. Below that, findings still have potential value but need additional manual verification. When using this report with Oracle Real Application Testing (its intended use), commonality is naturally very high.

Generating the Report

Single instance: DBMS_ADDM.COMPARE_INSTANCES. RAC: DBMS_ADDM.COMPARE_DATABASES. Pass begin/end snapshot IDs for both the base period and compare period.

Reading the Report

Check SQL commonality at the top first. Two load graphs appear in the middle — base period and compare period — each with a reference line for CPU core count. Three tabs below: Configuration (system changes detected), Findings (performance differences with regression/improvement flags and descriptions), Resource (CPU, memory, I/O, interconnect comparison side by side).

Part 10

Statspack

When to Use Statspack

Statspack predates AWR (introduced pre-10g) and remains available in Oracle 12c. Its single advantage: no separate license needed. The decision rule is simple — if you have an AWR license, use AWR and ignore Statspack. If you don't, Statspack is your primary snapshot and reporting tool.

Feature AWR Statspack
License Requires separate Diagnostic Pack license Included with standard DB license
ADDM ✓ Supported ✗ Not available
Active Session History ✓ Included ✗ Not included
Automatic snapshots ✓ Built-in ✗ Must build your own scheduler
Report format HTML and text Text only
Issue detection depth Broader — some issue types only visible via AWR Narrower

Installation

Not installed by default. Run spcreate.sql from $ORACLE_HOME/rdbms/admin. Creates a schema named PERFSTAT. Interactive mode prompts for tablespace, temp tablespace, and password. Batch mode: assign variables then run the script. To uninstall: run spdrop.sql as SYSDBA.

Taking Snapshots

All operations go through the STATSPACK package. Create a snapshot with STATSPACK.SNAP (procedure) or as a function (returns snapshot ID). Default level: 5. Valid levels: 0, 5, 6, 7, 10 — higher = more data, more space.

To use a different level for one snapshot only: pass i_snap_level. To also make it the new permanent default: also pass TRUE to i_modify_parameter. To change the default without creating a snapshot: use STATSPACK.MODIFY_STATSPACK_PARAMETER. Current settings live in STATS$STATSPACK_PARAMETER. Existing snapshots: query STATS$SNAPSHOT.

Space Consideration Higher snapshot levels consume more storage. Monitor PERFSTAT schema growth rate if running Statspack long-term.

Baselines

Create with STATSPACK.MAKE_BASELINE — pass begin/end snapshot IDs, or begin/end dates (it finds the nearest snapshots). The function form returns the count of snapshots marked. Baselines don't create separate objects — they set a flag on snapshot rows, exempting them from purging. To remove a baseline (without deleting data): STATSPACK.CLEAR_BASELINE. Once cleared, snapshots become eligible for normal purging.

Generating Reports

Instance reports (spreport.sql): equivalent of an AWR report — full instance health summary across a snapshot interval. Interactive or batch mode. Prompts for begin/end snapshot IDs and output filename. SQL reports (sprepsql.sql): drill into a specific SQL statement by hash value.

Read Statspack reports top-down, just like AWR — high-level load indicators first to locate the problem area, then related detail sections. Note: Statspack sometimes expresses wait percentages relative to call time (DB time + background elapsed time) rather than DB time. In practice, treat call time percentages the same way — relative significance is comparable.

Purging

Use STATSPACK.PURGE. Accepts a range of snapshot IDs, begin/end dates, a cutoff date, or a number of days. Baseline-flagged snapshots are immune — clear their baseline first if you want them purged. To flush all Statspack data without reinstalling: run sptrunc.sql. Never manually truncate PERFSTAT tables.

Automation

Statspack has no built-in scheduler. Use DBMS_SCHEDULER or the OS task scheduler. At minimum, automate snapshot creation and purging. Baselines can be manual, or automated if preferred. Also automate gathering optimizer statistics on the PERFSTAT schema itself.

Documentation Detailed Statspack documentation is in $ORACLE_HOME/rdbms/admin/spdoc.txt. Oracle's standard performance tuning guide covers it lightly — the file itself is the authoritative reference.
Part 11

AWR In Depth

What AWR Captures

The Automatic Workload Repository (AWR) is Oracle's central performance data store. Every hour by default (configurable), the MMON background process takes a snapshot of key V$ views and saves them into the SYSAUX tablespace under the SYS schema. These snapshots accumulate, giving you a historical timeline of database behaviour you can query at any time.

AWR captures: time model statistics, system statistics (V$SYSSTAT), wait event data, SQL statistics including execution plans, segment-level activity, system metrics, ASH samples, OS statistics, and memory advisor data. This breadth is what makes it so powerful — a single AWR report integrates all of these into a structured, readable format.

License AWR requires the Oracle Diagnostic Pack license (part of Oracle Database Enterprise Edition). Using AWR without this license violates your Oracle agreement.

Key AWR Configuration Parameters

SNAP_INTERVAL
How often AWR takes a snapshot. Default: 60 minutes. Reduce to 30 or 15 minutes on heavily monitored systems to gain finer-grained historical visibility. Change with DBMS_WORKLOAD_REPOSITORY.MODIFY_SNAPSHOT_SETTINGS.
RETENTION
How long snapshots are kept before being purged. Default: 8 days (40320 minutes). Increase for longer baselines, but monitor SYSAUX space growth. Baselines are exempt from purging regardless of this setting.
TOPNSQL
Number of top SQL statements captured per snapshot, ordered by various criteria (elapsed time, CPU, gets, reads, executions). Default: 30. Increase if you need visibility into more SQL statements in reports.

Taking Manual Snapshots

Force a snapshot at any time with DBMS_WORKLOAD_REPOSITORY.CREATE_SNAPSHOT(). This is useful when you want to capture the state immediately before or after a change, giving you a clean boundary for comparison rather than relying on the hourly schedule.

AWR Baselines

Baselines preserve a range of AWR snapshots indefinitely, exempt from the normal retention purge. They serve two purposes: preserving a known-good period for comparison, and providing a statistical reference for adaptive thresholds.

Three baseline types exist. A fixed baseline covers a specific historical period you define — for example, a week of peak-season performance you want to keep as a reference. A moving window baseline dynamically tracks the most recent N days of snapshots, always staying current. A template baseline lets you define a repeating schedule for automatic baseline creation.

Create fixed baselines with DBMS_WORKLOAD_REPOSITORY.CREATE_BASELINE, passing a name, start time, and end time. View all baselines in DBA_HIST_BASELINE.

Generating AWR Reports

Run awrrpt.sql (single instance) or awrrpti.sql (RAC, per-instance) from $ORACLE_HOME/rdbms/admin. The script prompts for report format (HTML recommended — far more readable), begin snapshot ID, end snapshot ID, and a filename. For comparing two periods, run awrddrpt.sql (the AWR Difference report).

Reading an AWR Report — Structure

AWR reports follow a consistent structure. Read them top to bottom — each section builds on the context established by the previous one.

1
Header

Instance name, DB version, host, snapshot IDs, elapsed time, DB time, CPUs. Establish the baseline: what was the total DB time for this period? What was the load (DB time / elapsed time × CPUs)?

2
ADDM Findings

Read these first after the header. They give you the machine-generated diagnosis before you dive into raw statistics.

3
Top 10 Foreground Events

The single most useful section for initial diagnosis. Shows the wait events consuming the most DB time, expressed as percentage of total DB time and total waits. The top event(s) tell you where to focus next.

4
SQL Statistics

Top SQL by elapsed time, CPU, buffer gets, disk reads, executions, and parse calls. Identifies the heaviest SQL statements. Cross-reference with the wait event section — a SQL appearing in both is almost certainly contributing to the problem.

5
Instance Activity / Wait Event Histograms / Memory / I/O

Detail sections that support and explain what you found in the top events and SQL sections. Read whichever is relevant to the top events you identified.

AWR SQL Report

For detailed statistics on a single SQL statement across an AWR period, run awrsqrpt.sql. It requires the SQL ID (not hash value — different from Statspack). Provides execution statistics, plan information, and wait events specific to that statement. Essential when a SQL appears at the top of an AWR report and you need to understand exactly what it's doing.

Key DBA_HIST Views

The AWR repository is accessible directly as DBA_HIST_* views. Important ones: DBA_HIST_SNAPSHOT (all snapshot metadata), DBA_HIST_SYSSTAT (system statistics history), DBA_HIST_SYSTEM_EVENT (wait event history), DBA_HIST_SQL_PLAN (stored execution plans), DBA_HIST_SQLSTAT (SQL statistics per snapshot), DBA_HIST_SEG_STAT (segment-level statistics), DBA_HIST_OSSTAT (OS statistics).

Part 12

Active Session History (ASH)

What ASH Is

ASH fills the gap between real-time V$ views (which show only the current moment) and AWR snapshots (which show hourly aggregates). Every second, Oracle samples all active sessions — those currently on CPU or waiting for a non-idle event — and records a row per session into an in-memory circular buffer: V$ACTIVE_SESSION_HISTORY. This buffer typically holds the last 30–60 minutes of samples, depending on session activity volume.

Every hour, a subset of these samples is flushed to DBA_HIST_ACTIVE_SESS_HISTORY, where they persist within the AWR retention period. The flush is not every sample — typically 1-in-10 is written to disk, which is sufficient for trend analysis while keeping storage manageable.

Why ASH is Powerful AWR tells you what happened in aggregate over an hour. ASH lets you zoom into a 5-minute window, a specific session, or a specific SQL statement and see exactly what was happening second by second. It's essential for diagnosing intermittent problems and short-duration incidents that don't show up clearly in hourly AWR data.

What Each ASH Row Contains

Each sample captures: session ID and serial number, SQL ID and plan hash, the current wait event name and class, the wait event's P1/P2/P3 parameters, blocking session (if any), the object being accessed, service name, module and action (application context set by DBMS_APPLICATION_INFO), and the sample timestamp. This richness means you can slice ASH data almost any way — by SQL, by session, by event, by object, by application module.

Practical ASH Queries

What was the database doing at a specific time?

Filter V$ACTIVE_SESSION_HISTORY to a narrow sample_time window and aggregate by event name. Each row represents one session-second of work. The count of rows per event is proportional to the time spent on that event during the window.

What SQL was causing the most activity?

Group by SQL_ID and count samples. High sample counts mean that SQL was frequently seen as active — it was consuming significant DB time during the analysis window.

Was there a blocking chain?

Join ASH to itself on blocking_session. Walk the chain from the blocked session up to the root blocker. The root blocker is the session that needs attention.

ASH Reports

Run ashrpt.sql (single instance) or ashrpti.sql (RAC) to generate a formatted ASH report for any time window. The report aggregates ASH data into top events, top SQL, top sessions, top blocking sessions, and top DB objects — similar in structure to AWR but focused on a narrow window and with per-second granularity. This is the first tool to reach for when investigating a short-duration incident.

ASH in EM Express / OEM

The Performance Hub in EM Express includes an ASH analytics interface — a visual activity timeline broken down by wait class, SQL, session, or service. This is often the fastest way to spot patterns in ASH data without writing SQL. OEM Cloud Control's ASH Analytics provides even more interactive filtering and drill-down capability.

Part 13

Memory Architecture & Tuning

Oracle Memory Structure Overview

Oracle uses two broad memory areas: the System Global Area (SGA), shared by all sessions connected to the instance, and the Program Global Area (PGA), private to each server process. Together these are sometimes called the instance memory. Getting memory allocation right is one of the highest-leverage tuning activities — too little causes physical I/O and parse failures; too much wastes OS memory that could be used for OS caching.

SGA Components

Component 01
Buffer Cache

Caches data blocks read from datafiles. The larger it is, the more blocks stay in memory and the fewer physical reads are needed. The primary I/O reduction lever. Sized by DB_CACHE_SIZE.

Component 02
Shared Pool

Holds the library cache (parsed SQL and PL/SQL) and the data dictionary cache. Too small causes hard parses and ORA-04031 errors. Sized by SHARED_POOL_SIZE.

Component 03
Large Pool

Used for RMAN backup/restore operations, parallel query message buffers, and shared server session memory. Keeping RMAN out of the shared pool prevents it from evicting library cache objects.


Component 04
Java Pool

Memory for Java session state and Java class definitions if the JVM is in use. Only relevant if the database runs Java stored procedures.

Component 05
Streams Pool

Used by Oracle Streams and GoldenGate capture/apply processes. Size this if replication is in use.

Component 06
Redo Log Buffer

Circular buffer holding redo entries before LGWR writes them to redo log files. Sized by LOG_BUFFER. Usually doesn't need tuning — LGWR writes very frequently. log buffer space waits indicate it's too small.

Automatic Memory Management

Oracle provides three tiers of memory management automation:

1
Manual — set each component individually

Set DB_CACHE_SIZE, SHARED_POOL_SIZE, etc. explicitly. Full control, full responsibility. Used when you understand exactly what the workload needs.

2
ASMM — Automatic Shared Memory Management

Set SGA_TARGET to the total desired SGA size. Oracle dynamically redistributes memory between SGA components based on workload demand. Individual component sizes become minimums (or 0 for fully automatic sizing). Enabled when SGA_TARGET > 0 and MEMORY_TARGET = 0.

3
AMM — Automatic Memory Management

Set MEMORY_TARGET to the total combined SGA + PGA size. Oracle manages both SGA and PGA automatically, shifting memory between them as needed. Simplest to configure; less predictable in some mixed workloads. Not available on all OS platforms (not supported on Linux with HugePages).

Buffer Cache Tuning

The key metric is the Buffer Cache Hit Ratio: (logical reads − physical reads) / logical reads × 100. A ratio below ~95% in an OLTP system suggests the buffer cache may be undersized, though this metric alone is insufficient — a full table scan will always cause physical reads regardless of cache size, and those scans shouldn't penalise the hit ratio calculation.

More useful: query V$DB_CACHE_ADVICE. This advisory view shows the estimated number of physical reads at various hypothetical cache sizes based on current workload, helping you determine whether adding memory would actually reduce I/O meaningfully.

The buffer cache uses an LRU (Least Recently Used) algorithm. Blocks accessed frequently stay in the cache; infrequently accessed blocks age out. Full table scans on large tables can flood the cache and evict useful blocks — Oracle uses a special algorithm to insert large-table scan blocks near the tail of the LRU list so they don't pollute the cache. The CACHE hint (or table property) forces blocks to the MRU end, useful for small frequently-scanned tables.

Shared Pool Tuning

The shared pool's library cache holds parsed and compiled SQL and PL/SQL. When a SQL statement is submitted, Oracle checks the library cache for a matching parsed version (a soft parse). If not found, it parses from scratch — a hard parse, which is CPU-intensive and requires latches that can become contention points. Hard parses also hold shared pool latches longer, causing waits for other sessions.

The primary weapon against hard parses is bind variables. SQL using literals (WHERE id = 1234) generates a unique SQL text for each value, causing a hard parse every time. SQL using bind variables (WHERE id = :b1) reuses the same cursor for all values. Monitor V$SQL — look for statements with high PARSE_CALLS relative to EXECUTIONS that have no bind variables.

Query V$SHARED_POOL_ADVICE for the shared pool equivalent of the cache advisor — estimated parse time savings at different pool sizes.

PGA Tuning

PGA is private per-session memory used for sort operations, hash joins, bitmap operations, and other work area operations. When a work area operation exceeds the available PGA memory, Oracle spills to temporary tablespace — a massive performance hit. PGA_AGGREGATE_TARGET sets a soft limit on total PGA usage across all sessions. PGA_AGGREGATE_LIMIT (12c+) sets a hard limit. Query V$PGA_TARGET_ADVICE for sizing recommendations.

Part 14

I/O Tuning

Understanding Oracle I/O

I/O is the most common performance bottleneck in Oracle databases. Oracle performs two broad categories of I/O: datafile I/O (reading and writing database blocks) and redo I/O (writing redo log entries). These have very different patterns and tuning approaches.

Datafile reads are driven by the buffer cache miss rate. Datafile writes are handled by DBWR (Database Writer) background processes, which write dirty blocks from the buffer cache to datafiles. Redo writes are handled by LGWR (Log Writer), which writes the redo log buffer to online redo log files on every commit and when the buffer reaches one-third full.

Key I/O Wait Events and Their Meaning

db file sequential read
Single-block read. Typical for index lookups, undo reads, control file reads. High occurrence usually means many index range scans or single-row lookups. Not inherently bad — but if dominating wait time, check if indexes are being used efficiently and whether reads are going to disk or being served from cache.
db file scattered read
Multi-block read (full table scan or fast full index scan). Each read fetches DB_FILE_MULTIBLOCK_READ_COUNT blocks. High values indicate frequent large table scans. Investigate missing indexes or whether full scans are intentional (e.g., large batch operations).
db file parallel read
Multiple non-contiguous blocks read in parallel during recovery or tablespace operations. Usually seen during instance startup or RMAN operations.
direct path read
Data read directly into PGA, bypassing the buffer cache. Seen during full table scans on large tables (Oracle may choose direct path for efficiency), parallel query, and RMAN. Not necessarily a problem — but high volumes on OLTP indicate large scan operations that may need indexes.
direct path write
Direct writes from PGA to disk, bypassing the buffer cache. Seen during sort operations that spill to temp, parallel DML, and direct-load operations. High values during OLTP indicate PGA pressure causing sort spills.
write complete waits
A process needs a buffer that DBWR is currently writing. Rare in healthy systems. High occurrence suggests I/O subsystem is struggling to keep up with DBWR write demand.

I/O Calibration

Oracle 11g+ includes DBMS_RESOURCE_MANAGER.CALIBRATE_IO, which benchmarks the I/O subsystem and populates DBA_RSRC_IO_CALIBRATE with maximum IOPS, maximum MB/s, and latency figures. This gives you a measured baseline of what your storage can actually deliver, which is essential for interpreting I/O wait times — 10ms latency means something very different on HDD vs SSD vs NVMe.

Striping and Storage Layout

Distribute datafiles across multiple physical storage paths (disk groups in ASM, multiple LUNs in traditional storage). Oracle ASM (Automatic Storage Management) handles striping and mirroring transparently and is the recommended storage layer for Oracle databases. ASM's AU_SIZE (allocation unit) and stripe width affect sequential read performance — larger values favour sequential workloads; smaller values favour random I/O.

Tuning DBWR

If free buffer waits appear frequently in the top wait events, DBWR is not writing dirty buffers fast enough to free space in the buffer cache. Solutions: increase the number of DBWR processes (DB_WRITER_PROCESSES), use async I/O (verify OS and storage support it via DISK_ASYNCH_IO=TRUE), or increase the I/O capacity of the storage. Multiple DBWR processes are especially beneficial on multi-CPU systems with many datafiles.

Part 15

SQL Execution Plans

What the Optimizer Does

The Cost-Based Optimizer (CBO) is Oracle's engine for choosing how to execute a SQL statement. For any given query, there may be dozens or hundreds of possible execution plans — different join orders, different access methods, different join algorithms. The optimizer estimates the cost (in units of I/O and CPU) of each candidate plan using statistics about the objects involved (row counts, column value distributions, index clustering factor) and selects the plan with the lowest estimated cost.

The optimizer's quality is only as good as its statistics. Stale or missing statistics cause bad plan choices. This is why DBMS_STATS maintenance is fundamental to stable performance.

Reading Execution Plans

Generate a plan with EXPLAIN PLAN FOR <sql> and view it with SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY). For plans of statements already in the cursor cache, use DBMS_XPLAN.DISPLAY_CURSOR(sql_id, child_number). For plans stored in AWR, use DBMS_XPLAN.DISPLAY_AWR(sql_id).

Plans are read from the innermost, topmost operation outward. The driving row source (the first operation to execute) is the most-indented line. Each operation shows: estimated rows (E-Rows), actual rows in runtime plans (A-Rows), cost, and bytes. Significant divergence between E-Rows and A-Rows is a red flag — the optimizer was wrong about cardinality, which usually cascades into a bad plan choice.

Common Plan Operations and What They Mean

TABLE ACCESS FULL
Full table scan. Reads every block in the table. Appropriate for large portions of a table (>10–20% of rows). Problematic when used for selective queries that should use an index. Investigate missing or unused indexes.
INDEX RANGE SCAN
Reads index entries within a range, then fetches the corresponding table rows. Appropriate for selective queries. High cost only when the range is too wide — may indicate statistics problems or a missing more selective index.
INDEX FULL SCAN / FAST FULL SCAN
Full scan reads the entire index in sorted order. Fast Full Scan reads the index in multiblock chunks (unordered) — faster but doesn't provide sorted output. Used when the index is smaller than the table and all needed columns are in the index (covering index).
NESTED LOOPS
For each row in the outer (driving) set, probe the inner set using an index. Excellent for joining small result sets or when the inner table has a selective index. Poor for large result sets — cost scales as outer_rows × inner_probe_cost.
HASH JOIN
Builds a hash table from the smaller (build) input, then probes it with the larger (probe) input. Excellent for joining larger result sets. Requires PGA work area for the hash table — if the build input is too large, it spills to temp.
SORT MERGE JOIN
Sorts both inputs then merges them. Useful when both inputs are already sorted (e.g., by index) or when neither sort nor hash join is efficient. Also spills to temp if sort memory is insufficient.

SQL Plan Baselines (SPM)

SQL Plan Management captures execution plans for SQL statements and stores them in a plan baseline. Once captured, the optimizer will only use a new plan for a statement if it has been verified to perform better than the baseline plan. This prevents plan regressions after statistics updates, parameter changes, or upgrades — a common cause of sudden performance degradation in production.

Plans are captured automatically (when OPTIMIZER_CAPTURE_SQL_PLAN_BASELINES=TRUE) or manually via DBMS_SPM.LOAD_PLANS_FROM_CURSOR_CACHE or DBMS_SPM.LOAD_PLANS_FROM_AWR. View baselines in DBA_SQL_PLAN_BASELINES.

Adaptive Query Optimization (12c+)

Oracle 12c introduced adaptive plans. The optimizer can defer certain plan decisions — particularly join method selection — until execution time, when it has seen actual row counts from the early plan steps. If actual rows differ significantly from estimates, the optimizer switches to a better join method mid-execution. This helps correct cardinality estimation errors without manual intervention, though it can occasionally produce unexpected plan instability.

Locking & Concurrency

Oracle's Locking Model

Oracle's concurrency model is fundamentally different from most other databases: readers don't block writers, and writers don't block readers. This is achieved through Multi-Version Concurrency Control (MVCC). When a query reads a row that has been modified but not yet committed, Oracle retrieves the pre-modification version of that row from the undo tablespace, giving the reader a consistent view of the data as it existed at the query's start time without needing to wait for the writer to commit.

However, writers do block writers. If session A has modified a row, session B attempting to modify the same row must wait until A commits or rolls back. This is a row-level lock — Oracle's DML locks are exclusively row-level, not page-level or table-level (except for DDL and specific lock modes).

Types of Locks

TX (Transaction)
Held for the duration of a transaction. Protects the rows modified by the transaction. All DML statements acquire TX locks. Released on COMMIT or ROLLBACK.
TM (DML Enqueue)
Table-level lock acquired during DML to prevent structural changes (DDL) to the table while a transaction is in progress. Not a data lock — doesn't conflict with other DML sessions on the same table.
DDL Locks
Exclusively acquired during DDL operations (ALTER TABLE, etc.). Conflict with all other DDL and with DML on the same object. Long-running DDL operations on busy tables cause widespread blocking.

Diagnosing Blocking Locks

When session B is blocked by session A, V$SESSION shows B's BLOCKING_SESSION column populated with A's SID. The EVENT column for B will be enq: TX - row lock contention (or similar enqueue wait). To find the root of a blocking chain, join V$SESSION to itself repeatedly on BLOCKING_SESSION until you reach a session with no blocker.

Query V$LOCK to see all currently held locks: TYPE, ID1, ID2, LMODE (lock mode held), and REQUEST (lock mode being requested). Sessions waiting for a lock have a non-zero REQUEST. Joining V$LOCK to V$SESSION and DBA_OBJECTS gives you a complete picture of who is blocking whom and on which object.

Common Cause Long-running uncommitted transactions are the most frequent cause of lock contention. Applications that open a transaction, perform DML, and then wait for user input before committing hold row locks for the entire wait duration — blocking anyone else who needs those rows.

Deadlocks

A deadlock occurs when session A holds a lock session B needs, and session B holds a lock session A needs — a circular wait. Oracle automatically detects deadlocks within a few seconds and resolves them by rolling back one statement (not the entire transaction) of the session chosen as the deadlock victim. The victim receives ORA-00060. A trace file is generated in the ADR (Automatic Diagnostic Repository) with the deadlock graph.

Deadlocks in application code almost always indicate a design flaw — typically, different code paths acquiring locks on the same set of rows but in different orders. Fix by ensuring all paths lock rows in a consistent order.

Latches and Mutexes

Below the lock layer are latches and mutexes — internal memory protection mechanisms. Unlike locks, they are not queued; a session attempting to acquire a held latch spins briefly, then sleeps. High latch contention appears in AWR as latch: shared pool, latch: library cache, or latch: cache buffers chains. These are symptoms of underlying problems — hard parses for library cache latches, hot blocks for cache buffers chains latches — not problems to fix directly.

Part 17

Redo & Undo

Redo: How Oracle Guarantees Durability

Every change made to Oracle data is first recorded in the redo log. Before a block can be considered changed, its change record must be written to redo. This write-ahead logging approach guarantees that committed changes can be recovered even if the database crashes immediately after the commit — the redo log contains everything needed to replay those changes against a consistent backup.

The flow: DML generates redo in the redo log buffer (in SGA). On commit, LGWR writes the buffer to the current online redo log file. Once written, the commit is confirmed to the application. Periodically (or when a log fills), Oracle switches to the next log file — a log switch. In archivelog mode, the filled log is then archived by the ARCn processes before it can be reused.

Redo Performance Problems

log file sync
A session waiting for LGWR to write its commit redo to disk. This is the commit latency. High values mean slow redo write I/O. Solutions: move redo logs to faster storage (SSD), increase log file size to reduce switch frequency, or investigate I/O contention on the redo log path.
log file parallel write
LGWR itself waiting for its I/O to complete when writing to redo logs. If this is high, the redo log storage is slow. Seen from LGWR's perspective; log file sync is seen from the committing session's perspective.
log buffer space
The redo log buffer is full and a session must wait for LGWR to free space. Usually means LGWR can't write fast enough, or the log buffer is undersized. Increase LOG_BUFFER or improve redo I/O performance.
log file switch (checkpoint incomplete)
Oracle wants to reuse a redo log file but DBWR hasn't finished writing all the dirty blocks associated with the previous checkpoint using that log. Add more redo log groups, increase redo log file size, or speed up DBWR.

Redo Log Sizing Guidelines

Redo log files should be sized so that log switches occur no more than 2–4 times per hour under normal load. Too-frequent switches cause checkpoint overhead and can cause log file switch (checkpoint incomplete) waits. Query V$LOG_HISTORY to see historical switch frequency. Use at least three redo log groups; for high-volume OLTP, more groups and larger files help.

Undo: How Oracle Maintains Read Consistency

When a session modifies a row, Oracle writes the before image of that row to the undo tablespace. This serves two purposes: it allows the change to be rolled back (ROLLBACK statement or instance crash recovery), and it provides the old version of the row to queries that need to see data as it existed before the change (read consistency / MVCC).

Undo data is stored in undo segments within the undo tablespace, managed automatically when UNDO_MANAGEMENT=AUTO (the default and recommended setting). The UNDO_RETENTION parameter specifies how long Oracle should try to retain undo data after a transaction commits — giving long-running queries time to use that undo for read consistency without hitting ORA-01555 (snapshot too old).

ORA-01555: Snapshot Too Old

This error occurs when a query needs an old version of a row for read consistency but that undo data has already been overwritten. It means the query ran long enough that Oracle's undo retention period wasn't sufficient to preserve the before-image. Solutions: increase UNDO_RETENTION, increase the undo tablespace size (so undo isn't overwritten even with longer retention), or investigate why the query takes so long.

Undo Tablespace Sizing

Size the undo tablespace based on: undo generation rate (visible in V$UNDOSTAT column UNDOBLKS), the configured UNDO_RETENTION, and your longest expected query duration. The UNDO_ADVISOR (accessible via OEM or DBMS_ADVISOR) can recommend an appropriate undo tablespace size based on actual workload data from AWR.

Oracle Tuning Advisors

Overview

Oracle provides a suite of advisors that automate analysis of specific performance areas and produce actionable recommendations. They all use the DBMS_ADVISOR framework internally and are accessible via EM Express/OEM or SQL*Plus. Most require the Diagnostic and Tuning Pack licenses.

Advisor 01
SQL Tuning Advisor

Analyzes specific SQL statements and recommends: gathering better statistics, creating SQL profiles (plan hints embedded in the data dictionary), restructuring queries, or creating indexes. Run on-demand or automatically by the automatic SQL tuning task (nightly, during the maintenance window).

Advisor 02
SQL Access Advisor

Recommends indexes, materialized views, and partitioning strategies for a given workload. Analyzes a set of SQL statements (from AWR, the cursor cache, or a custom workload) and suggests structural changes that would benefit the most statements.

Advisor 03
Memory Advisor

Actually three advisors: Buffer Cache Advisor (V$DB_CACHE_ADVICE), Shared Pool Advisor (V$SHARED_POOL_ADVICE), and PGA Advisor (V$PGA_TARGET_ADVICE). Each shows the estimated performance benefit of increasing or decreasing the respective memory area.


Advisor 04
Segment Advisor

Identifies segments (tables, indexes) that have significant wasted space due to row chaining, row migration, or deleted rows, and recommends shrink or rebuild operations. Run proactively or when investigating space issues.

Advisor 05
Undo Advisor

Recommends undo tablespace size based on undo generation rate (from V$UNDOSTAT) and the configured UNDO_RETENTION. Accessible via the DBMS_ADVISOR framework or OEM. Helps prevent ORA-01555 errors.

Advisor 06
Automatic SQL Tuning

Runs nightly during the maintenance window. Identifies high-load SQL from AWR, runs the SQL Tuning Advisor on them, and can automatically implement SQL profiles for statements with significant, verifiable improvement (when auto-implement is enabled).

Using the SQL Tuning Advisor Manually

Create a tuning task with DBMS_SQLTUNE.CREATE_TUNING_TASK, passing the SQL ID (for a statement from the cursor cache or AWR) or the SQL text directly. Execute with DBMS_SQLTUNE.EXECUTE_TUNING_TASK. Retrieve the report with DBMS_SQLTUNE.REPORT_TUNING_TASK. The report provides findings in three categories: statistics analysis, SQL profile recommendation, and index recommendations — each with an estimated performance improvement percentage.

On SQL Profiles A SQL profile stores corrected cardinality estimates and optimizer hints in the data dictionary, allowing the optimizer to choose a better plan without changing the SQL text. Unlike hints in the code, profiles persist across statistics refreshes and don't require application changes. Accept a profile with DBMS_SQLTUNE.ACCEPT_SQL_PROFILE.

Optimizer Statistics Management

Optimizer statistics — table row counts, column NDV (number of distinct values), histograms, index clustering factors — are gathered by DBMS_STATS. The automatic statistics gathering job runs nightly in the maintenance window, targeting objects whose statistics are stale (more than 10% of rows changed since last gather by default).

Key considerations: histograms are gathered for columns with skewed data distributions (where some values appear far more frequently than others) and are essential for the optimizer to generate accurate plans for such columns. The METHOD_OPT parameter in DBMS_STATS.GATHER_TABLE_STATS controls whether and how histograms are gathered — the default SIZE AUTO lets Oracle decide based on column usage monitoring (V$COL_USAGE_STATISTICS).

For volatile tables (frequently truncated and reloaded, or subject to large bulk inserts), consider locking statistics after a representative gather (DBMS_STATS.LOCK_TABLE_STATS) so the nightly job doesn't overwrite accurate statistics with stale post-load figures. Unlock before the next load cycle.

Pending Statistics Oracle 11g+ supports pending statistics — gathering statistics into a staging area and testing their effect before publishing them to the dictionary. Use DBMS_STATS.SET_TABLE_PREFS('schema','table','PUBLISH','FALSE') before gathering, then publish selectively with DBMS_STATS.PUBLISH_PENDING_STATS. This allows safe testing of statistics changes in production without risk.
