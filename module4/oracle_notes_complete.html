Part 01

Introduction & Tuning Areas

The Three Areas of Oracle Performance Tuning

Oracle performance tuning spans three intersecting areas. Performance Tuning Tools are the instruments DBAs reach for — V$ views, OS utilities, database-callable facilities, and third-party products. SQL Tuning involves writing efficient SQL, influencing the optimizer, and creating structures like indexes that help it. Instance Tuning addresses the database at the structural level: the SGA, disk I/O, background processes, client sessions, and the OS layer beneath it all.

Scope This course focuses on Performance Tuning Tools and Instance Tuning. SQL Tuning is not covered.

Performance Issue Attributes

Before diagnosing any issue, establish what you're dealing with by answering five questions:

Scope

Is the entire system affected, or just part of it? A full-system slowdown points to shared resources (CPU, I/O subsystem). A partial issue — one report taking 2 minutes, an ETL that grew from 3 hours to 12 — points somewhere specific.

Slow or Hanging?

A hanging situation — where the system appears frozen — is also a performance issue. The diagnosis path differs from a slow-response investigation.

Permanent or Intermittent?

Intermittent issues are harder to catch and diagnose. They require longer observation windows and often more invasive instrumentation.

Reproducible?

Reproducible issues are dramatically easier to trace — you can instrument and observe. Non-reproducible ones require relying on historical data.

Errors Involved?

Some errors are performance issues in disguise. A full archivelog destination hangs the database. An undersized shared pool triggers ORA-04031. Always check the alert log early.

Oracle's Performance Tuning Methodology

Because tuning touches every layer of the IT stack, there is no universal procedure — only a methodology. Two DBAs may follow different steps yet apply the same approach. This methodology begins after gathering issue attributes.

1
Set a measurable target

Ask how long the operation used to take. That baseline becomes your target. Without a target, you have no way to know when you're done.

2
Gather technical symptoms

The most important step — everything downstream depends on it. Use AWR reports, comparison AWR reports, V$ views, ADDM, the alert log, SQL tracing, and OS utilities. "Symptoms" here means technical indicators, not user descriptions.

3
List possible causes

Map the symptoms to known causes. For unfamiliar symptoms, consult Oracle Support documentation.

4
Develop a trial solution

Could be as simple as creating an index or following an ADDM recommendation. Could involve configuration or structural changes.

5
Test in a test environment

Mirror production as closely as possible. Use the same workload — Oracle's Database Replay tool can capture and replay production workloads on a test system.

6
Check the outcome

If the target is hit, stop. Further optimization beyond the target is a proactive development task, not tuning. If the target is missed, return to Step 3 with a different approach.

Emergency Exception When the entire system is unavailable — archivelog destination full, for instance — treat it as an emergency. Skip test environment validation and act immediately to restore service.

Describing System Workload

A DBA should be able to quantify workload in concrete terms for both tuning and stakeholder communication. Example: 100 users, one transaction each every 5 minutes, over an 8-hour day → 20 transactions/minute on average, 9,600/day. But workload isn't flat — always design for peak, not average. In this case, a reasonable design target might be 120 transactions/minute.

Part 02

IT Systems & Hardware

Server Performance Components

Oracle runs on top of an OS, which runs on top of hardware. Understanding the hardware layer is essential context for database tuning. The four components that govern server performance are CPU, memory, storage, and network. In production, you cannot change hardware specs — you monitor utilization and rate of consumption against the fixed capacity.

CPU

Performance is shaped by socket count, core count, cache size and level, clock speed, and processor architecture. More cores means more concurrent work can be handled — and typically higher Oracle licensing cost.

Memory

Type (DDR4, RDIMM, LRDIMM, NVDIMM) and transfer rate (MT/s) determine raw speed. The critical concern for a DBA: if the OS cannot find free memory, it swaps to disk — orders of magnitude slower. Monitor free memory closely.

Storage

Mechanical disks are a historic bottleneck; SSDs are far faster. Storage performance is measured in IOPS, latency, and throughput (MB/s). RAID configuration and connection type also affect effective performance. For multiple storage arrays, watch load distribution — an imbalanced setup creates a hidden bottleneck on the most-used path.

Network

Measured in bandwidth — 1 Gbps and 10 Gbps Ethernet are common. Utilization approaching 90% causes packet queuing. Persistent faulty packet reports signal a network health issue requiring investigation.

The IT Technology Stack

A typical system runs: database at the back, accessed by an application server via network, with clients at the front. Real deployments add firewalls, load balancers, and internet connections. A problem at any layer can surface as a performance issue at any other.

The most impactful layer for performance is application logic. Developers historically optimize for correctness, not efficiency. The classic example: a full table scan returning a few rows causes massive I/O contention for even a handful of users. A single well-chosen index transforms that into a few block reads. Most Oracle tuning targets exactly this — minimizing resource consumption per operation.

Course Scope This course focuses on the database layer. The goal is to identify whether a performance issue originates there, and if poorly written SQL is involved, isolate those statements for tuning or developer escalation.
Part 03

Performance Tuning Tools & Statistics

Oracle's Performance Tuning Toolset

V$ performance views live in memory and reset on instance restart — the most commonly used DBA diagnostic tool. Data dictionary views persist across restarts and live in tablespaces. AWR (Automatic Workload Repository) is the primary snapshot and reporting tool, but requires a separate license. Without it, use Statspack. ADDM proactively analyzes AWR data to identify and predict issues. Advisors provide targeted recommendations for specific areas. The alert log and trace files (auto-generated or manually triggered) round out the toolset. GUI access is available via EM Database Express (free) or OEM Cloud Control (licensed).

The Three Types of Performance Statistics

Type 01
Cumulative

Raw counts accumulated since the last instance startup. Reset on restart. Most are only meaningful compared over time. Some are intrinsically useful — enqueue waits should barely appear in a healthy system. Examples: time model, instance wait events.

Type 02
Metrics

The rate of change of a cumulative statistic, measured against time, transactions, or calls. The primary tool for detecting issues before they escalate. Power ADDM, AWR, and server alerts. E.g., database calls per second.

Type 03
Sampled

Statistics saved as periodic snapshots, enabling historical look-back. Lets you diagnose issues that occurred in the past. Main sources: ADDM, AWR, and Active Session History (ASH).

STATISTICS_LEVEL Parameter

Controls the depth of statistics Oracle collects. Three values:

BASIC
Disables most performance statistics and all automatic advisors. Use only for static-data, static-load databases where continuous collection adds no value.
TYPICAL
Default. Collects all major statistics, runs advisors on schedule. Sufficient for most production systems.
ALL
Adds timed OS statistics and execution plan statistics. Higher overhead. Used during active performance investigations. Can be set at session level — no need to change the whole instance when tuning one session.

To see which internal parameters each setting affects, query V$STATISTICS_LEVEL.

Part 04

The Time Model

DB Time, DB CPU & DB Wait Time

When no client request is being served, the database registers idle wait events. When a request arrives, the database consumes CPU, memory, and storage to process it. The time spent actively executing is DB CPU. Because resources are shared, the database often waits for a busy resource before proceeding — each such wait is a named wait event. The total of all non-idle waits is DB Wait Time.

These two together make DB Time — the total time the database spends actively serving client requests:

Formula DB Time = DB CPU + Total Non-Idle Wait Time

The Time Model is the set of statistics that classifies where DB time was spent. In Oracle 12c, it has roughly 20 categories — SQL execution elapsed time, hard/soft parse time, PL/SQL execution elapsed time, and others.

Important: DB Time Can Exceed Clock Time

DB time aggregates across all sessions. With 8 CPU cores, one real-time minute provides 8 CPU-minutes of capacity. If 10 sessions are each waiting simultaneously, all their wait times stack into DB time. A DB time of 2 hours in one real-time hour is perfectly normal on a busy, multi-core system.

Time Model Views

V$SYS_TIME_MODEL — system-wide accumulated times since last startup, in microseconds, organized hierarchically with DB time at the top. A second tree alongside it tracks Background Elapsed Time (excluded from DB time, but included to capture RMAN's impact since it's DBA-initiated and resource-intensive).

V$SESS_TIME_MODEL — same structure at the individual session level. Use it to identify which sessions consume the most DB time or DB CPU. Note: terminated sessions are not visible.

Why DB Time Matters

DB time is the single best indicator of database workload. It rises with load and falls when the database is idle. Beyond that, it enables four key diagnostic uses:

1
Measure overall load

If DB time is growing, the database is receiving more work. If it's flat, workload is stable.

2
Assess wait ratio

If total wait time is 50%+ of DB time, the system is spending more time waiting than working. This is a red flag for a poorly performing system.

3
Measure change impact

Added memory? Tuned a query? DB time before and after quantifies the real impact of the change.

4
Spot operation anomalies

If hard parses represent an unusually high share of DB time, that's worth investigating further — it's not normal in a well-tuned system.

Measuring Scalability with DB Time

As user load increases, both DB CPU and wait time should rise in proportion — that's a scalable system. At some point, wait time starts growing much faster than CPU time. That inflection is where users begin noticing degradation. Example data point: at 10 users, waits = 18% of DB time. At 50 users, waits = 47% (wait time +900s, CPU +160s). At 100 users, waits = 70% — clearly unacceptable.

Historical Data: DBA_HIST_SYS_TIME_MODEL

V$SYS_TIME_MODEL accumulates since startup and can't tell you what happened in a specific window. Oracle snapshots it hourly into DBA_HIST_SYS_TIME_MODEL. While queryable directly, the easiest way to read it is through an AWR report, which consolidates most DBA_HIST views into one readable document.

Part 05

Instance Activity & Wait Events

Instance Activity Views

While the time model gives high-level DB time breakdowns, instance activity views provide granular counters for hundreds of specific operations. The primary view is V$SYSSTAT — instance-level statistics accumulated since startup. It holds a statistic name and its cumulative value. Statistics are identified by name (not number — numbers can change across Oracle versions, names don't). Commonly watched: parse time CPU, physical reads, user commits.

To explore what's available with descriptions, query V$STATNAME. For service-level stats: V$SERVICESTATS. For session-level: V$SESSTAT (join to V$STATNAME for names). For the current session specifically: V$MYSTAT. For object-level detail including wait events broken down by segment: V$SEGMENT_STATISTICS — this one includes statistic names natively, so no join needed.

Wait Events

A wait event is a named record of the database waiting for a specific resource. There are over 600. The real-time diagnostic views are V$SESSION_WAIT and V$SESSION, which show events currently held by active sessions.

Reading Wait Columns

STATE
WAITING — currently waiting. WAITED UNKNOWN TIME — just finished, duration unknown (set TIMED_STATISTICS=TRUE). WAITED SHORT TIME — under 1/100s. WAITED KNOWN TIME — finished, duration in WAIT_TIME.
WAIT_TIME
Positive = last wait duration in 1/100s. 0 = still waiting. -1 = under 1/100s. -2 = timing unavailable.
SECONDS_IN_WAIT
Current wait duration in seconds. Only meaningful when STATE = WAITING.
P1, P2, P3
Event-specific parameters (object, file, block, or lock ID). Names in P1TEXT, P2TEXT, P3TEXT. Documented in Oracle Reference Appendix C.
Quick Rule Session is WAITING? → look at SECONDS_IN_WAIT. Wait has finished? → look at WAIT_TIME.

Common Wait Events

Buffer busy waits — process waiting for a data block in the buffer cache to become available. Free buffer waits — buffer cache is full. DB file sequential read — single-block read (index lookup, etc.). DB file scattered read — multi-block read (full table scan, etc.). Enqueue waits — serializing concurrent access to a resource. Library cache waits — pinning/locking an object in the library cache. Log buffer space — redo log buffer full. Log file sync — waiting for redo write to complete.

Key Principle Wait events are symptoms, not causes. They point toward the root cause but are not the root cause themselves. Some belong at the top of the wait list in a healthy system. Others should never appear there — their presence signals a real problem. Learning which is which is a major part of this course.
Part 06

Metrics & Event Histograms

What Metrics Tell You That Statistics Don't

A cumulative statistic just accumulates upward — it shows total volume but not pace or timing. A metric shows the rate of change of that statistic over a short interval. A graph of accumulated physical reads looks like a ramp. The same data as a metric shows exactly when reads spiked and by how much. That's the diagnostic value.

Most metrics are time-based (per second), but some use other dimensions (per transaction, per session). Metrics feed ADDM, AWR, and server alert systems. The full catalog is in V$METRIC_NAME. Oracle computes metrics for system, session, file, and wait event statistics.

Metric Views

V$-prefixed metric views live in memory, covering the last hour, updated every 15–60 seconds (varies by metric). When an AWR snapshot is taken, these are copied to their DBA_ counterparts with a SNAP_ID added. V$SYSMETRIC_HISTORY → DBA_HIST_SYSMETRIC_HISTORY. Not all V$ metrics are copied to DBA_ views. In normal operations, you rarely query these directly — tools consume them for you — but you may reference them to verify tool outputs.

Event Histograms

A metric spike (say, physical reads per second) could mean normal load growth or a system problem. Histograms let you distinguish the two. They bucket wait event occurrences by duration — each bucket represents a time range, and its value is how many waits fell there. Buckets double: <1ms, 1–2ms, 2–4ms, 4–8ms, 8–16ms, 16–32ms, 32–64ms…

If your storage normally delivers I/O in ~10ms and the histogram shows most waits concentrated in the 32–64ms bucket — something is wrong with the storage subsystem. If most waits sit in the 8–16ms bucket — that's close to normal, and the spike is likely just higher user load or a batch job starting.

Histogram Views

V$EVENT_HISTOGRAM — ~148 events, in milliseconds. V$EVENT_HISTOGRAM_MICRO — same data in microseconds. DBA_HIST_EVENT_HISTOGRAM — AWR-backed historical version. V$FILE_HISTOGRAM — single-block reads per data file.

Practical Note AWR reports present histogram data in tabular format, which is easier to read than querying V$ views directly. Know how to interpret histograms; you usually won't need to query them manually.
Don't Confuse These are wait event histograms. Optimizer statistics histograms are a completely separate concept used in SQL tuning — not relevant here.
Part 07

Server Generated Alerts

Proactive Alerting

Oracle's server generated alerts fire automatically when a metric crosses a predefined threshold, or when a specific event occurs. Threshold-based examples: physical reads/sec, user commits/sec, SQL service response time. Event-based examples: snapshot too old, resumable session suspended, recovery area space usage.

Two severity levels exist. A warning fires at the lower threshold — early notice. A critical fires at the upper threshold — needs immediate attention. Manage alerts via OEM (easiest) or programmatically via the DBMS_SERVER_ALERT package.

Setting Thresholds: DBMS_SERVER_ALERT.SET_THRESHOLD

METRIC_ID
The metric to monitor. Full list in PL/SQL Packages Reference, Table 143-3.
WARNING_OPERATOR
Comparison operator for warning threshold. Uses package constants for =, >, >=, <, <=, !=, list, or DO_NOT_CHECK (disables warning).
WARNING_VALUE
The warning threshold value. NULL = no warning alert.
CRITICAL_OPERATOR / CRITICAL_VALUE
Same pattern, for the critical level.
OBSERVATION_PERIOD
Minutes over which metric is computed and checked. Default: 1 minute.
CONSECUTIVE_OCCURRENCES
How many consecutive violations before the alert fires. Default: 1. Increase to tolerate expected brief spikes — e.g., short session blocking in a heavy-load app.
OBJECT_TYPE / OBJECT_NAME
Scope the threshold to a specific object type (system, file, service, tablespace, session) and named object. Constants in Table 143-1 of the documentation.

Default Thresholds & Retrieving Alert Info

Out of the box, only one threshold is active: tablespace space usage — warning at ≥85%, critical at >97%. No performance metrics are monitored by default. Query DBA_THRESHOLDS to see configured thresholds. Use DBMS_SERVER_ALERT.GET_THRESHOLD to retrieve a specific metric's threshold programmatically.

For generated alerts: DBA_OUTSTANDING_ALERTS (active, uncleared), DBA_ALERT_HISTORY (cleared), DBA_ALERT_HISTORY_DETAIL (both).

Best Practices

No universal thresholds exist — every system behaves differently. Build your baseline by reading AWR reports repeatedly while the system is healthy, then configure thresholds that flag meaningful deviations. For critical systems, connect alerts to email or SMS. If you have OEM, use its alerting system instead of the PL/SQL package.

Batch Processing Trap Thresholds set for normal OLTP workload will fire false alarms during batch windows. Oracle recommends adaptive thresholds for metrics like this — they learn from system behavior patterns and adjust automatically. Covered in a later lecture.
Remember Server alerts supplement your monitoring — they don't replace it. OS-level metrics and statistics outside Oracle's standard alert framework still need separate monitoring.
Part 08

ADDM — Automatic Database Diagnostic Monitor

What ADDM Does

ADDM reads AWR snapshots for a target period — and in-memory views where needed — to detect existing performance issues and predict ones likely to emerge soon. It then produces findings and recommendations. By default it runs automatically after each AWR snapshot, analyzing the interval between the two most recent. It can also be invoked manually across any snapshot range. All results are stored in the AWR.

ADDM's performance indicator is DB time. Every finding and recommendation aims at reducing DB time for the detected bottleneck. ADDM performs analysis in two dimensions simultaneously: user-request operations (connection time, SQL execution, user calls) and resource metrics (CPU, memory, I/O, database locks). In both dimensions it does cause-and-effect analysis — symptoms first, then root cause.

License Required ADDM requires the same license as AWR. No AWR license = no ADDM.

Accessing ADDM

Via EM Express

Performance Hub → ADDM tab. Lists tasks by time period. Each task shows finding count and average active sessions. Click a task to see actionable findings and informational findings. The informational panel tells you what's not a problem — useful for narrowing scope. Each actionable finding shows its DB time impact percentage, recommendations, rationale, and expected benefit.

In the AWR Report

ADDM appears as the first section after the header, and again as a full copy at the report's end. Findings are ordered by percentage impact on average active sessions. Average active sessions = DB time ÷ elapsed time (both in the report header).

When reading: check dependencies before acting on order. If findings include top SQL, high CPU usage, and undersized SGA — investigate CPU and SGA first. A slow SQL statement may simply be a symptom of the SGA being too small. Fix the root cause, generate a fresh AWR report, and confirm the downstream findings have cleared.

From SQL*Plus

Use DBMS_ADVISOR.GET_TASK_REPORT(task_name). Task names come from data dictionary views. Output is text only — no HTML — but well structured and readable.

Perspective ADDM is a tool, not a performance expert. It identifies areas of concern. Validating its recommendations and understanding their dependencies is your job as a DBA.

Running ADDM Manually

Required when the default two-snapshot window is too narrow. Condition: no instance restart within the analysis period.

A
DBMS_ADVISOR package

Create a task → set begin/end snapshot IDs via SET_TASK_PARAMETER → execute with EXECUTE_TASK → retrieve with GET_TASK_REPORT.

B
Run addmrpt.sql (simpler)

Prompts for begin/end snapshot IDs and a report filename. Generates and saves automatically. Use addmrpti.sql for RAC.

Part 09

ADDM Comparison Report

Purpose

The ADDM comparison report measures the performance impact of a specific change — a parameter tweak, index creation, hardware upgrade, or any planned intervention. Unlike the AWR comparison report (which surfaces statistical deltas for the DBA to interpret), the ADDM comparison report attempts the interpretation for you, linking detected system changes to observed performance differences via cause-and-effect analysis.

Base Period vs Compare Period

The base period covers AWR snapshots from before the change — the normal system state. The compare period covers snapshots after the change, whether that represents a regression or an improvement. ADDM identifies configuration changes (SGA size, parameters), measures the performance delta, and tries to map changes to outcomes.

SQL Commonality

SQL commonality measures how similar the SQL workload is between the two periods (0–100%). The higher, the more reliable the comparison. Treat findings as reliable only at 80%+ commonality. Below that, findings still have potential value but need additional manual verification. When using this report with Oracle Real Application Testing (its intended use), commonality is naturally very high.

Generating the Report

Single instance: DBMS_ADDM.COMPARE_INSTANCES. RAC: DBMS_ADDM.COMPARE_DATABASES. Pass begin/end snapshot IDs for both the base period and compare period.

Reading the Report

Check SQL commonality at the top first. Two load graphs appear in the middle — base period and compare period — each with a reference line for CPU core count. Three tabs below: Configuration (system changes detected), Findings (performance differences with regression/improvement flags and descriptions), Resource (CPU, memory, I/O, interconnect comparison side by side).

Part 10

Statspack

When to Use Statspack

Statspack predates AWR (introduced pre-10g) and remains available in Oracle 12c. Its single advantage: no separate license needed. The decision rule is simple — if you have an AWR license, use AWR and ignore Statspack. If you don't, Statspack is your primary snapshot and reporting tool.

Feature AWR Statspack
License Requires separate Diagnostic Pack license Included with standard DB license
ADDM ✓ Supported ✗ Not available
Active Session History ✓ Included ✗ Not included
Automatic snapshots ✓ Built-in ✗ Must build your own scheduler
Report format HTML and text Text only
Issue detection depth Broader — some issue types only visible via AWR Narrower

Installation

Not installed by default. Run spcreate.sql from $ORACLE_HOME/rdbms/admin. Creates a schema named PERFSTAT. Interactive mode prompts for tablespace, temp tablespace, and password. Batch mode: assign variables then run the script. To uninstall: run spdrop.sql as SYSDBA.

Taking Snapshots

All operations go through the STATSPACK package. Create a snapshot with STATSPACK.SNAP (procedure) or as a function (returns snapshot ID). Default level: 5. Valid levels: 0, 5, 6, 7, 10 — higher = more data, more space.

To use a different level for one snapshot only: pass i_snap_level. To also make it the new permanent default: also pass TRUE to i_modify_parameter. To change the default without creating a snapshot: use STATSPACK.MODIFY_STATSPACK_PARAMETER. Current settings live in STATS$STATSPACK_PARAMETER. Existing snapshots: query STATS$SNAPSHOT.

Space Consideration Higher snapshot levels consume more storage. Monitor PERFSTAT schema growth rate if running Statspack long-term.

Baselines

Create with STATSPACK.MAKE_BASELINE — pass begin/end snapshot IDs, or begin/end dates (it finds the nearest snapshots). The function form returns the count of snapshots marked. Baselines don't create separate objects — they set a flag on snapshot rows, exempting them from purging. To remove a baseline (without deleting data): STATSPACK.CLEAR_BASELINE. Once cleared, snapshots become eligible for normal purging.

Generating Reports

Instance reports (spreport.sql): equivalent of an AWR report — full instance health summary across a snapshot interval. Interactive or batch mode. Prompts for begin/end snapshot IDs and output filename. SQL reports (sprepsql.sql): drill into a specific SQL statement by hash value.

Read Statspack reports top-down, just like AWR — high-level load indicators first to locate the problem area, then related detail sections. Note: Statspack sometimes expresses wait percentages relative to call time (DB time + background elapsed time) rather than DB time. In practice, treat call time percentages the same way — relative significance is comparable.

Purging

Use STATSPACK.PURGE. Accepts a range of snapshot IDs, begin/end dates, a cutoff date, or a number of days. Baseline-flagged snapshots are immune — clear their baseline first if you want them purged. To flush all Statspack data without reinstalling: run sptrunc.sql. Never manually truncate PERFSTAT tables.

Automation

Statspack has no built-in scheduler. Use DBMS_SCHEDULER or the OS task scheduler. At minimum, automate snapshot creation and purging. Baselines can be manual, or automated if preferred. Also automate gathering optimizer statistics on the PERFSTAT schema itself.

Documentation Detailed Statspack documentation is in $ORACLE_HOME/rdbms/admin/spdoc.txt. Oracle's standard performance tuning guide covers it lightly — the file itself is the authoritative reference.
