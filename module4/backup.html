⌨️ RMAN Command Language
Stand-alone commands, RUN blocks, and persistent configurations

🔹 Stand-alone Commands

Executed immediately at RMAN prompt

RMAN> BACKUP DATABASE;
RMAN> BACKUP
  2> DATABASE
  3> ;
# Comments with hash symbol

📦 RUN Blocks (Job Commands)

Commands executed sequentially as a single unit

RMAN> RUN {
  2>   ALLOCATE CHANNEL c1 DEVICE TYPE DISK;
  3>   BACKUP DATABASE;
  4>   RELEASE CHANNEL c1;
  5> }
⚠️ If any command fails → RMAN stops the entire block

⚙️ Persistent RMAN Settings

Stored in control file (and optionally recovery catalog) - persist across sessions

-- View all settings SHOW ALL; -- View specific setting SHOW CONTROLFILE AUTOBACKUP FORMAT; -- Change setting CONFIGURE PARALLELISM 3 FOR DEVICE TYPE sbt_tape; -- Reset to default CONFIGURE PARALLELISM CLEAR;
-- Query from SQL SELECT * FROM V$RMAN_CONFIGURATION;

2. Batch Mode

📈 Incremental Backups & Block Change Tracking
Save time, space, and I/O with smarter backups

📊 Level 0 Backup

🔵

Full backup of ALL database blocks

Forms the base for all Level 1 backups

⚠️ Level 0 ≠ Full backup (Full = all used blocks, but Level 0 is the base for incrementals)

📈 Level 1 Backup

🟡

Backs up ONLY changed blocks

Can only be backupsets (not image copies)

⚖️ Differential vs Cumulative Incremental Backups

🔹 DIFFERENTIAL (Default)

Backs up: Changes since last incremental (Level 1 or Level 0)

Size: ✅ Smaller

Recovery: RMAN reads all Level 1 backups + Level 0

Best for: Saving storage space

🔸 CUMULATIVE

Backs up: All changes since last Level 0 backup

Size: ❌ Larger

Recovery: RMAN reads only most recent Level 1 + Level 0

Best for: Faster recovery (fewer files to read)

📌 Recommendation: Differential backups are preferred in large databases to save disk space.
-- Differential incremental (default) BACKUP INCREMENTAL LEVEL 1 DATABASE; -- Cumulative incremental BACKUP INCREMENTAL LEVEL 1 CUMULATIVE DATABASE;

⚡ Block Change Tracking (BCT)

Speeds up incremental backups by tracking changed blocks

How it works

  • Oracle maintains a binary BCT file with addresses of changed blocks
  • RMAN uses this file instead of scanning all blocks
  • BCT file grows in 10 MB increments - remains small even for heavy workloads

Limitations

  • Not suitable if >20% of database changes frequently
  • Tracks up to 8 incremental backups
  • Not enabled by default
-- Enable BCT ALTER DATABASE ENABLE BLOCK CHANGE TRACKING USING FILE '/u01/oradata/bct_file.bct'; -- Disable BCT ALTER DATABASE DISABLE BLOCK CHANGE TRACKING; -- Move BCT file ALTER DATABASE RENAME BLOCK CHANGE TRACKING FILE '/old/bct.bct' TO '/new/bct.bct'; -- Monitor BCT SELECT file#, status, blocks, bytes, file_name FROM V$BLOCK_CHANGE_TRACKING;

✅ Best Practices for Incremental Backups

  1. Take Level 0 incremental backup on weekends (Note: Level 0 ≠ full backup)
  2. Take Level 1 incremental backups during weekdays (off-peak hours)
  3. Prefer differential over cumulative for smaller backup size
  4. Enable Block Change Tracking (BCT) for large databases
  5. After 7 incremental backups, take a new Level 0

⏱️ Backup Retention Policy
What to keep, what to mark obsolete

🔁 REDUNDANCY-BASED

3

Keeps last N backups

CONFIGURE RETENTION POLICY TO REDUNDANCY 3; -- Keeps last 3 backups -- Older backups marked OBSOLETE

Default: REDUNDANCY 1

📅 WINDOW-BASED

7

Days of recovery window

CONFIGURE RETENTION POLICY TO RECOVERY WINDOW OF 7 DAYS; -- Can recover to any point in last 7 days -- Common in production

⚠️ CRITICAL WARNINGS

  • Only ONE retention policy can be active at a time
  • CONFIGURE RETENTION POLICY TO NONE - Not recommended, especially with FRA
  • For incrementally updated backups: Keep redundancy at 1, use UNTIL TIME in RECOVER
-- Disable retention policy (NOT RECOMMENDED) CONFIGURE RETENTION POLICY TO NONE; -- Reset to default (REDUNDANCY 1) CONFIGURE RETENTION POLICY CLEAR;
🆘 RMAN Recovery: RESTORE vs RECOVER
The most important distinction in RMAN

🔄 RESTORE

"Bring files FROM backup"

  • ✅ Creates database files from backup sets or image copies
  • ✅ Restores file to backup point, NOT failure time
  • ✅ Can restore datafiles, control files, SPFILE, archived redo logs
  • ❌ Cannot restore temporary tablespaces
RESTORE DATABASE; RESTORE TABLESPACE users; RESTORE DATAFILE 5; RESTORE CONTROLFILE; RESTORE SPFILE;

⚡ RECOVER

"Apply changes TO reach desired point"

  • ✅ Applies archived redo logs and incremental backups
  • ✅ Brings datafile to specific SCN/time
  • ✅ Complete recovery: up to failure point
  • ✅ Incomplete/PITR: up to specified time before failure
RECOVER DATABASE; RECOVER TABLESPACE users; RECOVER DATAFILE 5; RECOVER DATABASE UNTIL TIME '2024-01-01:12:00:00';

📌 RESTORE Requirements

  • DB can be started, mounted, or open
  • If DB open: file to restore must be offline
  • Most recent backup by default

📌 RECOVER Requirements

  • ⚠️ Database must be in ARCHIVELOG mode!
  • Cannot recover in NOARCHIVELOG mode

🔄 Recovery Scenarios

💿 Whole Database Recovery

-- NOARCHIVELOG mode STARTUP MOUNT; RESTORE DATABASE; RECOVER DATABASE; -- typically does nothing ALTER DATABASE OPEN RESETLOGS; -- ARCHIVELOG mode (complete) STARTUP MOUNT; RESTORE DATABASE; RECOVER DATABASE; ALTER DATABASE OPEN;

📁 User Tablespace Recovery

-- Database OPEN ALTER TABLESPACE users OFFLINE; RESTORE TABLESPACE users; RECOVER TABLESPACE users; ALTER TABLESPACE users ONLINE;

✅ Does not affect SYSTEM, SYSAUX, UNDO, TEMP

📦 Recovery to New Location

RUN { SET NEWNAME FOR DATAFILE 5 TO '/new_location/users01.dbf'; RESTORE DATAFILE 5; SWITCH DATAFILE ALL; -- Update control file RECOVER DATAFILE 5; }
⚡ Recovery by Switching Data Files
The fastest recovery method using image copies

✨ Concept: SWITCH instead of RESTORE

Instead of restoring from backup sets, update control file to point to existing image copies

🔥 SCENARIO 1: All Database Datafiles Lost

Disk X failed - database shut down

-- 1. Mount database STARTUP MOUNT; -- 2. Switch to image copies SWITCH DATABASE TO COPY; -- 3. Apply changes RECOVER DATABASE; -- 4. Open ALTER DATABASE OPEN;

✅ No actual data transfer - only control file updated!

🔧 SCENARIO 2: Single User Tablespace Datafile Lost

Database remains OPEN

-- 1. Take datafile offline ALTER DATABASE DATAFILE 5 OFFLINE; -- 2. Switch to image copy SWITCH DATAFILE 5 TO COPY; -- 3. Apply changes RECOVER DATAFILE 5; -- 4. Bring online ALTER DATABASE DATAFILE 5 ONLINE;

✅ Zero downtime for other tablespaces

💡 Advantages of Using Image Copies

  • 🚀 Faster than restoring from backup sets - no data movement, just metadata update
  • 📉 Minimal downtime - especially for large databases
  • 🔄 Works for complete or partial recovery
  • 💪 Can be used for standby database setup

⚠️ Image copies on separate storage should have similar performance specs as production disks to avoid application degradation.

⏰ Point-in-Time Recovery (PITR) & Incarnations
Rewinding the database to a specific moment

🎯 PITR Overview

Purpose: Restore database to a previous state

Common use: Accidental data changes, missing archive logs

Recovery point options:

  • 🔢 SCN (System Change Number)
  • 📋 Redo log sequence number
  • 📌 Restore point
  • ⏲️ Timestamp

Internal: Oracle converts all to SCN

⚠️ REQUIREMENTS:
  • ARCHIVELOG mode
  • Backups of all datafiles before recovery SCN

📋 DBPITR Procedure

RUN { -- Set recovery point SET UNTIL TIME '2024-01-15:14:30:00'; -- or UNTIL SCN 123456 -- or UNTIL SEQUENCE 1234 RESTORE DATABASE; RECOVER DATABASE; } -- Must open with RESETLOGS! ALTER DATABASE OPEN RESETLOGS;
📌 DRAWBACKS:
  • Entire database unavailable during recovery
  • Time-consuming (restores all datafiles)

🔄 Database Incarnation

Each RESETLOGS operation creates a new database incarnation

⚠️ CRITICAL:

  • Archived redo logs and database must have matching incarnations for recovery
  • Incarnation affects which backups can be restored

💾 DBID (Database Identifier):

  • Required if restoring SPFILE or control file
  • Found in autobackup filename, RMAN logs
  • Best practice: Save DBID externally with backup files

🔍 Open Read-Only Before RESETLOGS

Useful when exact recovery point is unclear

-- 1. Shutdown SHUTDOWN IMMEDIATE; -- 2. NOMOUNT, set DBID STARTUP NOMOUNT; SET DBID 1234567890; -- 3. Restore control file from autobackup RESTORE CONTROLFILE FROM AUTOBACKUP; -- 4. Mount and recover to point ALTER DATABASE MOUNT; RUN { SET UNTIL TIME '2024-01-15:14:30:00'; RESTORE DATABASE; RECOVER DATABASE; } -- 5. Verify data ALTER DATABASE OPEN READ ONLY; -- 6. If satisfied, shutdown and OPEN RESETLOGS SHUTDOWN IMMEDIATE; STARTUP MOUNT; ALTER DATABASE OPEN RESETLOGS;

⚠️ This method overwrites the current control file!

🔍 Validation & Preparation for Recovery
Always check before you restore

👁️ RESTORE PREVIEW

RESTORE DATABASE PREVIEW;

Shows backup files RMAN will access

⚠️ Does NOT check physical presence

✅ RESTORE VALIDATE HEADER

RESTORE DATABASE VALIDATE HEADER;

Validates backup file headers

Checks if backups are in correct location

🛡️ VALIDATE

VALIDATE DATABASE; VALIDATE TABLESPACE users;

Checks for corrupted blocks in backups

⏱️ Can be skipped in urgent scenarios

📋 Recovery Preparation Checklist

  • ✅ Identify files that need recovery
  • ✅ Determine DBID (if restoring control file/SPFILE)
  • ✅ Validate backups availability
  • ✅ Check backup integrity
  • ✅ Restore archived redo logs if needed
  • ✅ Provide password for encrypted backups

⚠️ Common RMAN Mistakes & Troubleshooting
Learn from others' errors - what NOT to do

❌ CRITICAL MISTAKES

  • 🗑️ Deleting archived logs manually
    → RMAN repository out of sync, recovery failures
  • 💾 No retention policy
    → FRA fills up, database hangs
  • 🔧 NOARCHIVELOG mode in production
    → No point-in-time recovery, data loss guaranteed
  • 📏 Never testing recoveries
    → Backups might be corrupt when you need them

✅ SOLUTIONS

  • 🔄 Always use RMAN to delete
    DELETE ARCHIVELOG, DELETE OBSOLETE
  • 📊 Configure retention policy
    REDUNDANCY 2 or RECOVERY WINDOW 7 DAYS
  • 🔥 Enable ARCHIVELOG mode
    Required for ALL production databases
  • 🧪 Test recovery quarterly
    Validate backups, practice PITR

🔧 Common RMAN Errors & Fixes

RMAN-06059: Expected archived log not found

→ Backup or delete archived logs without updating RMAN repository

✅ RUN { CROSSCHECK ARCHIVELOG ALL; DELETE EXPIRED ARCHIVELOG ALL; }

ORA-19815: FRA is full

→ No retention policy, backups not marked obsolete

✅ CONFIGURE RETENTION POLICY; DELETE OBSOLETE;

RMAN-03002: Failure during recovery

→ Incarnation mismatch, wrong UNTIL time

✅ LIST INCARNATION; RESET DATABASE TO INCARNATION;

ORA-00205: Control file not found

→ Lost control file, need autobackup restore

✅ SET DBID; RESTORE CONTROLFILE FROM AUTOBACKUP;
