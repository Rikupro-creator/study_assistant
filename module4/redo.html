Complete Visual Guide to Redo Management, ARCHIVELOG Mode, and Best Practices

📚 Quick Reference: Redo Log Essentials
Purpose, Architecture & Critical Concepts

🎯 PURPOSE OF REDO LOGS

Redo logs record ALL changes made to the database.

  • Redo entries are generated whenever data changes
  • First stored in memory (Redo Log Buffer)
  • Later written to disk (Online Redo Log Files)
⚠️ CRITICAL: If online redo logs are missing or deleted → database will crash

🔄 REDO LOG ARCHITECTURE

1. User DML → Redo entry generated
2. Redo Log Buffer ← Stored in memory (SGA)
3. LGWR → Writes to Online Redo Logs

Redo logs are written sequentially, not randomly

🎯 The Golden Rule of Redo

Redo = Forward recovery (replays changes)
Undo = Backward recovery (reverses changes)

🔄 Redo Log Groups & Circular Writing
Understanding the sequential, circular nature of redo logs
2 Production best practice: 3+ groups File sizes: Fixed, do not auto-grow
🎯 Key Concepts - Redo Log Groups:
  • Redo log groups: Each group contains one or more identical members (multiplexing)
  • Minimum groups required: 2 (database cannot run with less)
  • Production best practice: 3 or more groups
  • Circular reuse: LGWR writes sequentially, then wraps around - this is NORMAL behavior
  • LOG SWITCH: When current group fills, LGWR moves to next group
🛡️ Redo Log Multiplexing
Protecting against disk failure with multiple members per group
💡 Redo Multiplexing Best Practices:
  • Each group should have at least 2 members on separate physical disks
  • Place members on different storage controllers when possible
  • Oracle does NOT know disk topology - DBA must decide member placement
  • Even with storage RAID/ASM, most production systems still multiplex redo logs (conservative best practice)
📀 NOARCHIVELOG vs ARCHIVELOG Mode
The single most important decision for database recoverability

❌ NOARCHIVELOG

TEST/DEV ONLY

Characteristics:

  • ❌ No point-in-time recovery
  • ❌ Cannot recover to past state
  • ❌ Cannot use Data Guard or advanced HA
  • ✅ Only instance recovery
  • ✅ No media recovery capability
⚠️ What happens: Old redo entries are OVERWRITTEN during reuse

✅ ARCHIVELOG

PRODUCTION STANDARD

Characteristics:

  • ✅ Point-in-time recovery
  • ✅ Can recover to past state
  • ✅ Data Guard & HA features
  • ✅ Online backups possible
  • ⚠️ Must manage archive space
📦 What happens: ARCn copies full redo logs BEFORE reuse → Archived Redo Logs ⚠️ CRITICAL: Oracle does NOT manage archive space - DBA must delete old archive logs!

⚠️ CRITICAL: Archive Destination Failure

If all archive destinations are disabled while in ARCHIVELOG mode:

🚫 DATABASE WILL HANG During log switches until a valid destination is set!

Best Practice: Always configure multiple archive destinations with ALTERNATE failover.

⚙️ Redo Log Management Commands
Essential DBA commands for redo log and archive management

➕ Adding Redo Log Groups

-- Add new group with automatic group number ALTER DATABASE ADD LOGFILE ('/disk1/redo04.log', '/disk2/redo04.log') SIZE 1G; -- Add new group with explicit group number ALTER DATABASE ADD LOGFILE GROUP 4 ('/disk1/redo04.log', '/disk2/redo04.log') SIZE 1G; -- Add member to existing group (must be INACTIVE) ALTER DATABASE ADD LOGFILE MEMBER '/disk3/redo01b.log' TO GROUP 1;
⚠️ RULE: Cannot modify CURRENT redo log group!

➖ Dropping Redo Log Groups

-- Drop entire group (must be INACTIVE) ALTER DATABASE DROP LOGFILE GROUP 3; -- Drop individual member ALTER DATABASE DROP LOGFILE MEMBER '/disk1/redo01.log'; -- Force log switch to make group INACTIVE ALTER SYSTEM SWITCH LOGFILE; -- Check group status first! SELECT group#, status FROM v$log;
📌 RULES:
  • Database must have at least 2 groups
  • Cannot drop last member of a group
  • In ARCHIVELOG: group must be archived first

📦 Relocating Redo Log Members

-- Step 1: Shutdown database SHUTDOWN IMMEDIATE; -- Step 2: Copy files at OS level -- $ cp /old_disk/redo01.log /new_disk/redo01.log -- Step 3: Startup MOUNT STARTUP MOUNT; -- Step 4: Rename in control file ALTER DATABASE RENAME FILE '/old_disk/redo01.log' TO '/new_disk/redo01.log'; -- Step 5: Open database ALTER DATABASE OPEN;

⚠️ Original files must be available and valid during rename!

📀 ARCHIVELOG Management

-- Enable ARCHIVELOG mode SHUTDOWN IMMEDIATE; STARTUP MOUNT; ALTER DATABASE ARCHIVELOG; ALTER DATABASE OPEN; -- Disable ARCHIVELOG mode SHUTDOWN IMMEDIATE; STARTUP MOUNT; ALTER DATABASE NOARCHIVELOG; ALTER DATABASE OPEN; -- Check current mode ARCHIVE LOG LIST; -- or SELECT log_mode FROM v$database;

📍 Archive Destination Configuration

-- Modern method: LOG_ARCHIVE_DEST_n (up to 31) ALTER SYSTEM SET LOG_ARCHIVE_DEST_1='LOCATION=/u01/archive'; ALTER SYSTEM SET LOG_ARCHIVE_DEST_2='LOCATION=/u02/archive'; ALTER SYSTEM SET LOG_ARCHIVE_DEST_3='LOCATION=/u03/archive'; -- With FRA (preferred) ALTER SYSTEM SET LOG_ARCHIVE_DEST_1='USE_DB_RECOVERY_FILE_DEST'; -- With ALTERNATE failover ALTER SYSTEM SET LOG_ARCHIVE_DEST_2='LOCATION=/u02/archive MAX_FAILURE=1 ALTERNATE=3'; ALTER SYSTEM SET LOG_ARCHIVE_DEST_3='LOCATION=/u03/archive'; ALTER SYSTEM SET LOG_ARCHIVE_DEST_STATE_3=ALTERNATE;
-- LOG_ARCHIVE_FORMAT (naming) -- Include %s %t %r for unique names ALTER SYSTEM SET LOG_ARCHIVE_FORMAT='arch_%t_%s_%r.arc'; -- Variables: -- %s = sequence number -- %t = thread number (RAC) -- %r = resetlogs ID -- %d = database ID -- NOTE: Ignored if using FRA
🔄 Redo vs Undo - The Complete Picture
Understanding the fundamental difference and how they work together

🔴 REDO

"How do I REAPPLY what happened?"

➡️ MOVES DATA FORWARD

📋 Purpose:

  • ✅ Media recovery - restore from backup
  • ✅ Instance crash recovery - roll forward
  • ✅ Replication - Data Guard, Streams

💾 Storage:

  • 📁 Online Redo Log Files
  • 📦 Archived Redo Log Files (if ARCHIVELOG)

🟢 UNDO

"How do I REVERSE what happened?"

⬅️ MOVES DATA BACKWARD

📋 Purpose:

  • ✅ ROLLBACK operations - undo changes
  • ✅ READ CONSISTENCY - consistent query results
  • ✅ FLASHBACK operations - view past data

💾 Storage:

  • 📁 UNDO Tablespaces
  • 📂 Undo Segments

🔄 Instance Crash Recovery: Redo + Undo Together

PHASE 1
🔴 ROLL FORWARD

SMON reads redo logs and reapplies ALL changes

Uses REDO logs

PHASE 2
🟢 ROLL BACK

SMON reads undo and reverses uncommitted transactions

Uses UNDO data

Together they guarantee: Database consistency and complete recoverability

✅ Redo Log Best Practices
Production-tested guidelines from real-world DBA experience

🔢 Number of Groups

3-4
  • Minimum: 2 groups
  • Production: 3 groups minimum
  • Recommended: 4 groups if storage allows
  • Why: Prevents waits, smoother log switching

📏 Redo Log Size

1GB+
  • Guideline: Log switches every 15-20 minutes
  • Too frequent: Increase size (more frequent than 15 min)
  • Batch jobs: Occasional 10-min switches OK
  • DBCA default: 200MB (too small for OLTP!)

⚡ Storage Performance

FASTEST AVAILABLE
  • Redo logs written continuously by LGWR
  • High latency = Poor performance
  • Priority: Redo logs > Datafiles
  • Use: SSD, NVMe, high-speed SAN

🛡️ Multiplexing

  • 2+ members per group on separate disks
  • Different controllers if possible
  • Even with RAID/ASM: Still multiplex
  • Conservative = Safe

📀 ARCHIVELOG Mode

  • ALL production databases must be in ARCHIVELOG
  • NOARCHIVELOG: Test/DEV only
  • Multiple destinations with ALTERNATE
  • Monitor space! Oracle doesn't auto-delete

💾 Archive Storage

  • High capacity - not performance critical
  • Use FRA for automatic organization
  • Implement deletion policy (after backup)
  • RMAN: DELETE ARCHIVELOG ...

🎯 The DBA's Redo Log Commandments

1️⃣ Never delete or modify online redo logs - database will crash

2️⃣ Always multiplex - protect against disk failure

3️⃣ Production = ARCHIVELOG - no exceptions

4️⃣ Size redo logs for 15-20 min switches - not too frequent

5️⃣ Fastest storage for redo logs - performance critical

6️⃣ Monitor archive destinations - avoid database hang

7️⃣ Multiple archive destinations - with ALTERNATE failover

8️⃣ Delete archive logs after backup - Oracle won't do it for you

📊 Essential Dynamic Performance Views
Query these views to monitor and troubleshoot redo logs

📋 V$LOG

Redo log group information

SELECT group#, sequence#, bytes/1024/1024 size_mb, status, archived, first_time FROM v$log ORDER BY group#; -- STATUS values: -- CURRENT → Currently being written -- ACTIVE → Needed for recovery, not current -- INACTIVE → Available for reuse

📁 V$LOGFILE

Redo log member information

SELECT group#, member, status, type FROM v$logfile ORDER BY group#, member; -- STATUS: INVALID = inaccessible -- STATUS: STALE = incomplete -- STATUS: DELETED = removed -- NULL = normal

📍 V$ARCHIVE_DEST

Archive destination status

SELECT dest_name, status, binding, destination, error FROM v$archive_dest WHERE status != 'INACTIVE';

📦 V$ARCHIVED_LOG

Archived redo log history

SELECT sequence#, name, blocks, completion_time, deleted FROM v$archived_log WHERE completion_time > SYSDATE - 7;

🔍 Common Diagnostic Queries

-- Check if archive destination is full SELECT dest_name, error, target, destination, fail_sequence FROM v$archive_dest WHERE error IS NOT NULL; -- Find log switch frequency (last 24h) SELECT TO_CHAR(first_time, 'YYYY-MM-DD HH24') hour, COUNT(*) switches FROM v$log_history WHERE first_time > SYSDATE - 1 GROUP BY TO_CHAR(first_time, 'YYYY-MM-DD HH24') ORDER BY 1; -- Check ARCn processes SELECT * FROM v$archive_processes;

📌 Key Points for Multitenant:

  • Redo logs belong to the CDB - shared by all PDBs
  • Must connect to CDB$ROOT to manage redo logs
  • Cannot manage redo logs from a PDB - will get ORA-65040
  • Redo generation is higher in multitenant OLTP systems
  • Redo log groups are typically large (multi-GB)
⚠️ Common Mistakes & Troubleshooting
Learn from others' errors - what NOT to do

❌ CRITICAL MISTAKES

  • 🗑️ Deleting online redo logs manually
    → Database crashes immediately
  • 💾 Running out of archive space
    → Database HANGS, no log switches possible
  • 🔧 Modifying CURRENT redo log
    → ORA-01623, operation not allowed
  • 📏 Redo logs too small
    → Excessive log switches, performance issues

✅ SOLUTIONS

  • 🔄 Always use ALTER DATABASE commands
    → Never touch redo files at OS level
  • 📊 Monitor archive space
    → Set up alerts, delete after backup
  • ⚡ Force log switch first
    → ALTER SYSTEM SWITCH LOGFILE;
  • 📈 Size for 15-20 min switches
    → Add larger groups, drop small ones

🆘 Recovery Scenarios

Lost redo log member (multiplexed)

-- Database continues running! -- Just drop and add new member ALTER DATABASE DROP LOGFILE MEMBER '/lost/redo01.log'; ALTER DATABASE ADD LOGFILE MEMBER '/new/redo01.log' TO GROUP 1;

Lost all members of INACTIVE group

-- Drop the whole group ALTER DATABASE DROP LOGFILE GROUP 3; -- Recreate it ALTER DATABASE ADD LOGFILE GROUP 3 ('/disk1/redo03.log', '/disk2/redo03.log') SIZE 1G;
⚠️ If CURRENT group is lost: Instance crash, requires incomplete recovery (lose data). This is why multiplexing is CRITICAL!
