⚠️ Important: PDB State After Creation

Initial State: MOUNTED

• Newly created PDB is in MOUNT mode
• Users CANNOT connect yet
• Must manually OPEN it first
• Use: ALTER PLUGGABLE DATABASE OPEN;
Exception: DBCA

• DBCA automatically opens PDB
• Ready to use immediately
• Convenience feature
• SQL*Plus requires manual open

3. CREATE PLUGGABLE DATABASE Statement - Complete Syntax
All important clauses and options explained
CREATE PLUGGABLE DATABASE pdb_name
-- Source Options:
  FROM PDB$SEED | existing_pdb | remote_pdb

-- File Location (choose one):
  FILE_NAME_CONVERT = ('/source/', '/dest/')
  CREATE_FILE_DEST = '/u01/oradata/pdb' -- 12.2+
  -- Or rely on DB_CREATE_FILE_DEST (OMF)

-- Additional Options:
  DEFAULT TABLESPACE users
    DATAFILE '/u01/oradata/pdb/users01.dbf'
    SIZE 100M AUTOEXTEND ON

  STORAGE (MAXSIZE 10G) -- Limit PDB size

  ROLES = (role1, role2) -- Grant to PDB_DBA

  NO DATA -- Clone structure only, not data
;

📂 File Location Options (Priority Order)

1. FILE_NAME_CONVERT (Highest Priority)
• Maps source paths to destination paths
• String replacement in filenames
• Example: '/seed/' → '/pdb1/'
• Most explicit control
2. CREATE_FILE_DEST (12.2+)
• Overrides CDB's OMF setting
• PDB-specific OMF location
• Auto-generates filenames
• Good for one-off changes
3. DB_CREATE_FILE_DEST (Default)
• CDB-level OMF parameter
• Used if nothing else specified
• Auto-managed filenames
• ✅ Recommended approach!
4. PDB_FILE_NAME_CONVERT
• Parameter-based mapping
• Takes effect if clause not used
• Similar to FILE_NAME_CONVERT
• Less commonly used

⚙️ Important Additional Clauses

DEFAULT TABLESPACE
• Creates new tablespace
• Sets as default for PDB
• ✅ Highly recommended!
• Keeps SYSTEM clean
STORAGE (MAXSIZE)
• Limits PDB total size
• Prevents runaway growth
• Good for multi-tenant
• Example: MAXSIZE 10G
ROLES
• Grants roles to PDB_DBA
• Local role in PDB
• Given to admin user
• More on roles later
NO DATA
• Clones structure only
• No actual data copied
• Fast clone creation
• Useful for templates
💡 DBA Recommendation - Use OMF!

OMF (Oracle Managed Files) is the preferred approach:

  • Automatic Naming: Oracle generates unique filenames automatically
  • Less Error-Prone: No manual path/filename mistakes
  • Easier Management: No need to track individual file locations
  • Best Practice: Set DB_CREATE_FILE_DEST at CDB level, let Oracle handle the rest
  • GUID-Based Paths: Uses PDB GUID for unique directories (guaranteed uniqueness)
⚠️ Important Notes:
  • Not all clauses are covered here - only the ones you need at this stage
  • If DB_CREATE_FILE_DEST is set (OMF enabled), you can omit FILE_NAME_CONVERT entirely
  • CREATE_FILE_DEST was introduced in 12.2 - not available in 12.1
  • NO DATA clause is useful when you want schema definitions but not the actual data
4. Creating PDB from SEED - Complete Workflow
Most common method - starting from PDB$SEED template
🎯 Key Points - Creating from SEED:
  • Simplest Method: Just one statement if OMF is configured: CREATE PLUGGABLE DATABASE pdbname;
  • SEED is Template: PDB$SEED (CON_ID=2) provides base structure - SYSTEM, SYSAUX tablespaces
  • Initial State MOUNTED: New PDB cannot accept connections until you manually OPEN it
  • SET TIMING ON: Useful to see how long creation takes (typically 20-30 seconds)
  • Easy Connect: Format is user@//host[:port]/service_name.domain - port 1521 is default
  • OMF Recommended: Let Oracle manage filenames - uses GUID for unique directory names
💡 Workflow Summary:
  1. Connect to CDB as common user (SYS/SYSTEM)
  2. Run CREATE PLUGGABLE DATABASE statement
  3. PDB is created in MOUNTED state
  4. Run ALTER PLUGGABLE DATABASE OPEN to make it available
  5. Test connection using easy connect method
  6. Verify status with SELECT name, open_mode FROM v$pdbs;
5. Dropping a Pluggable Database
Cleanup process - must close before dropping
📝 Drop PDB Syntax Options:
-- Drop and delete files (most common) DROP PLUGGABLE DATABASE pdbname INCLUDING DATAFILES; -- Drop but keep files (for plug-in later) DROP PLUGGABLE DATABASE pdbname; -- Drop with all options DROP PLUGGABLE DATABASE pdbname INCLUDING DATAFILES KEEP DATAFILES; -- Contradictory - use one or the other!
🔑 Key Concepts:
  • Two-Step Process: Always CLOSE first, then DROP - cannot drop while OPEN
  • INCLUDING DATAFILES: Permanently deletes physical files from disk
  • Without Including: PDB metadata removed but files remain (useful for plug-in scenarios)
  • Irreversible: No "undr op" command - only backup/restore can recover
  • Quick Operation: Dropping is much faster than creating (seconds vs minutes)

⚠️ Common Issues & Solutions

Problem: "PDB not found"
Solution: You're not connected to ROOT. Connect as SYS to ROOT container.
Problem: "Cannot connect to PDB"
Solution: PDB is still MOUNTED. Run ALTER PLUGGABLE DATABASE OPEN.
Problem: "Cannot drop PDB"
Solution: PDB is still OPEN. Close it first before dropping.
Problem: "Insufficient privileges"
Solution: User needs CREATE PLUGGABLE DATABASE privilege.

Remember:

  • New PDB state: MOUNTED (must manually OPEN)
  • Exception: DBCA auto-opens PDB
  • OMF is your friend - use DB_CREATE_FILE_DEST
  • Always create default tablespace

Prerequisites:

  • CDB must be OPEN (READ WRITE)
  • Need common user (SYS/SYSTEM)

Best Practices:

  • Enable OMF for easier file management
  • Document PDB admin credentials
  • Test connection after creation
  • Close before dropping PDBs
  • INCLUDING DATAFILES when dropping
